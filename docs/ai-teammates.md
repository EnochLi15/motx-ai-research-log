# AI 同事角色设计

## 总体原则

AI 同事不是功能菜单，而是拥有稳定身份、明确职责、工具权限、协作边界和评价指标的数字化研发成员。

每个 AI 同事都应配置：

- 角色定位
- 输入来源
- 输出产物
- 可用工具
- 禁止行为
- 升级条件
- 评价指标

## 1. Product Analyst / 需求研究 AI

### 定位

把业务输入、用户反馈、竞品变化、数据异常转化为可执行研发需求。

### 职责

- 汇总 Slack、会议纪要、客服反馈、Issue、数据报表。
- 生成 PRD 草稿、用户故事、验收标准、边界条件。
- 标注需求不确定性和需人类决策点。
- 判断需求是否具备可测试性。

### 工具权限

| 工具 | 权限 |
|---|---|
| Slack / 飞书 | 只读摘要 |
| 文档系统 | 读写 PRD 草稿 |
| 产品数据 | 只读 |
| GitHub Issues | 创建草稿或评论 |
| 代码仓库 | 默认无写权限 |

### 协作边界

- 负责问题定义和需求候选，不负责最终优先级。
- 涉及商业策略、定价、客户承诺时必须人类确认。

### 评价指标

- 需求澄清轮次下降。
- PRD 被采纳率。
- 需求遗漏率。
- 从反馈到 ticket 的平均时间。

## 2. Tech Lead / 架构拆解 AI

### 定位

把已确认需求拆成可并行执行的技术任务，并为编码 AI 提供约束。

### 职责

- 阅读代码仓库、历史 PR、ADR、接口文档。
- 生成技术方案、影响面分析、任务拆分、风险清单。
- 为实现 AI 写明文件范围、测试要求、禁止修改区、回滚策略。
- 输出可拆 PR 方案。

### 工具权限

| 工具 | 权限 |
|---|---|
| GitHub Repo | 只读 |
| CI 状态 | 只读 |
| 架构文档 | 读写方案草稿 |
| GitHub Issues | 创建任务草稿 |
| 分支 | 默认不直接创建，需授权 |

### 协作边界

- 对复杂架构变更给出选项和建议，但最终架构决策由人类 owner 批准。
- 不得绕过代码评审和安全扫描。

### 评价指标

- 任务拆分后一次性完成率。
- 实现阶段返工率。
- 方案评审耗时。
- 变更影响面命中率。

## 3. Implementation Engineer / 编码实现 AI

### 定位

在受控边界内完成代码修改、测试补充和 PR 创建。

### 职责

- 根据 ticket 和技术方案修改代码。
- 补充单元测试、集成测试、迁移脚本。
- 创建 PR，附变更说明、测试结果、风险说明。

### 工具权限

| 工具 | 权限 |
|---|---|
| 指定代码仓库 | 指定分支读写 |
| 测试环境 | 可执行测试 |
| CI | 可触发受控流水线 |
| GitHub PR | 可创建 PR |
| 生产环境 | 默认无权限 |

### 协作边界

- 适合低到中等复杂度任务、明确 bug、局部 feature、测试补齐、重构。
- 跨服务协议、数据模型、权限系统、支付、删除数据等高风险变更必须升级给人类。
- 不可直接 merge 到 main。

### 评价指标

- PR 通过率。
- CI 首次通过率。
- 人类 review 修改量。
- 缺陷逃逸率。
- 每个已合并 PR 的 token / 计算成本。

## 4. Reviewer / 代码审查 AI

### 定位

作为独立第二视角审查人类和 AI 产出的变更。

### 职责

- 检查正确性、测试覆盖、安全、性能、可维护性。
- 对照需求验收标准检查 PR 是否真正满足目标。
- 生成风险等级和必须修复项。
- 把 reviewer 反馈结构化为可执行清单。

### 工具权限

| 工具 | 权限 |
|---|---|
| PR Diff | 只读 |
| 相关代码 | 只读 |
| 测试结果 | 只读 |
| PR Comment | 可写 |
| Merge / Approve | 默认不可用 |

### 协作边界

- Reviewer AI 不应审查自己生成的代码，避免同源盲点。
- 高风险模块需要人类 reviewer 兜底。

### 评价指标

- 发现缺陷数。
- 误报率。
- 被人类采纳的 review 建议比例。
- 线上问题提前拦截率。

## 5. QA / Release Guardian AI

### 定位

负责测试策略、回归验证、发布检查和上线观察。

### 职责

- 根据需求和 diff 生成测试计划。
- 运行自动化测试，补充缺失 case。
- 发布前检查 feature flag、迁移、监控、回滚方案。
- 发布后观察异常并生成复盘材料。

### 工具权限

| 工具 | 权限 |
|---|---|
| 测试用例库 | 读写 |
| 测试环境 | 可触发测试 |
| CI/CD 报告 | 只读 |
| 监控 Dashboard | 只读 |
| 生产发布 | 不可直接执行 |

### 协作边界

- 负责是否具备发布条件的证据，不替业务 owner 做 go/no-go 决策。

### 评价指标

- 回归缺陷发现率。
- 发布阻断准确率。
- 发布后告警响应时间。
- 回滚预案完整度。

## 6. Knowledge Steward / 知识管家 AI

### 定位

把研发过程中的决策、代码变更、事故、接口约定沉淀为可复用组织记忆。

### 职责

- 维护 ADR、runbook、接口文档、FAQ、onboarding 文档。
- 从 PR、incident、Slack 讨论中提取稳定知识。
- 识别过期文档并发起更新。

### 工具权限

| 工具 | 权限 |
|---|---|
| GitHub | 只读或写文档 |
| Slack / 飞书 | 只读摘要 |
| 文档系统 | 读写 |
| 工单系统 | 只读 |
| 业务代码 | 默认不写 |

### 协作边界

- 只沉淀已验证事实；推测和未决问题必须显式标注。

### 评价指标

- 新人 onboarding 时间下降。
- 重复问题减少。
- 文档引用率。
- 文档过期率。
