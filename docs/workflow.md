# 软件研发 AI 原生作业范式

## 1. 传统流程的问题

传统软件研发流程往往高度依赖人类串行推进：

```text
需求输入 → 人类澄清 → 人类拆任务 → 人类写代码 → 人类 Review → 人类测试 → 人类发布 → 人类复盘
```

主要瓶颈：

- 上下文反复解释。
- 需求与实现脱节。
- Review 堵塞。
- 测试后置。
- 文档滞后。
- 经验无法复利。

## 2. AI 原生流程

AI 原生流程强调：人类设目标，AI 并行产出证据，人类做关键判断。

```text
目标输入
→ 需求研究 AI 澄清需求
→ 架构拆解 AI 输出方案和任务图
→ 编码实现 AI 在分支执行
→ 代码审查 AI 做第一轮 Review
→ QA AI 生成测试与发布证据
→ 人类 Owner 审批关键节点
→ 知识管家 AI 沉淀规则和复盘
```

## 3. 端到端流程

### 阶段 1：需求进入

输入可能来自：

- 用户反馈
- 客服工单
- 业务目标
- 产品规划
- Slack / 飞书讨论
- 数据异常
- 竞品变化

输出：

- 需求摘要
- 用户价值
- 验收标准
- 待澄清问题
- 初始风险

### 阶段 2：方案拆解

Tech Lead AI 根据需求生成：

- 技术方案
- 影响面分析
- 依赖关系
- 任务拆分
- PR 拆分建议
- 测试要求
- 风险等级

### 阶段 3：上下文包生成

每个任务都需要上下文包：

```text
任务目标
相关需求
验收标准
相关代码路径
可修改文件
禁止修改文件
测试要求
团队规则
升级条件
```

### 阶段 4：AI 编码执行

编码 AI 在隔离分支执行：

- 修改代码
- 补测试
- 跑 lint / test / build
- 创建 PR
- 写明变更说明、测试证据、风险说明

### 阶段 5：AI 初审

Reviewer AI 检查：

- 是否满足验收标准
- 是否有安全风险
- 是否测试充分
- 是否修改范围过大
- 是否违反团队规则
- 是否需要拆 PR

### 阶段 6：人类 Review

人类只重点关注：

- 架构取舍
- 产品语义
- 高风险模块
- 权限与安全
- 发布时机
- 最终批准

### 阶段 7：QA 与发布

QA / Release Guardian AI 生成：

- 回归测试清单
- 测试执行报告
- 发布检查清单
- 监控项
- 回滚方案

### 阶段 8：知识沉淀

Knowledge Steward AI 更新：

- ADR
- Runbook
- FAQ
- 接口文档
- 团队规则
- 已知坑
- 复盘文档

## 4. 关键机制

### 4.1 任务对象化

每个任务必须成为可验证对象，而不是一句聊天指令。

### 4.2 PR 证据化

每个 PR 应包含：

- 背景
- 需求映射
- 变更摘要
- 测试证据
- 风险说明
- 回滚方案

### 4.3 Review 结构化

Review 评论应分类：

```text
must_fix
suggestion
discussion
risk
question
```

### 4.4 反馈规则化

重复出现的问题要转化为规则：

```text
问题：API 修改未更新文档
规则：凡修改 /api/** 必须更新 OpenAPI 和 SDK 文档
```

## 5. 人机分工

| 工作 | AI 负责 | 人类负责 |
|---|---|---|
| 需求整理 | 初稿、澄清问题、验收标准 | 价值判断、优先级 |
| 技术方案 | 草案、影响面、风险 | 架构拍板 |
| 编码 | 明确任务实现 | 高复杂设计、关键模块 |
| Review | 第一轮扫描 | 最终判断 |
| 测试 | 用例、回归、报告 | 高风险验证 |
| 发布 | 检查清单、证据 | Go / No-Go |
| 文档 | 自动沉淀 | 重要决策确认 |

## 6. 总结

AI 原生研发作业范式不是让 AI 完全替代工程师，而是让 AI 承担可结构化、可验证、可并行的研发劳动，让人类聚焦目标、判断、取舍和责任。
