# 研发 AI 同事效率与评价指标体系

## 1. 设计原则

不能只用单一 benchmark 衡量软件研发 AI。同一个 AI 在 SWE-bench 上表现好，不代表它在真实研发组织中能稳定创造价值。

评价体系应覆盖：

- 速度
- 质量
- 成本
- 协作
- 风险
- 学习能力

## 2. 总体指标

| 指标 | 含义 |
|---|---|
| Lead Time | 从需求进入到 PR 合并的时间 |
| Cycle Time | 从任务开始到完成的时间 |
| PR Merge Rate | AI 生成 PR 的合并率 |
| CI First Pass Rate | 首次 CI 通过率 |
| Review Rounds | 平均 Review 轮次 |
| Human Modification Ratio | 人类修改占比 |
| Defect Escape Rate | 缺陷逃逸率 |
| Cost Per Accepted PR | 每个有效 PR 的 AI 成本 |
| Context Rebuild Time | 上下文重建时间 |
| Documentation Coverage | 文档沉淀覆盖率 |

## 3. 按 AI 同事分类指标

### Product Analyst

- 需求澄清轮次
- PRD 被采纳率
- 验收标准完整度
- 需求遗漏率
- 从反馈到 ticket 的平均时间

### Tech Lead

- 任务拆分准确率
- 方案评审通过率
- 实现阶段返工率
- 影响面命中率
- 高风险问题提前发现率

### Implementation Engineer

- PR 通过率
- CI 首次通过率
- 平均实现时间
- 人类 review 修改量
- 缺陷逃逸率
- 每个已合并 PR 的成本

### Reviewer

- 缺陷发现数
- 被采纳建议比例
- 误报率
- 高风险问题识别率
- 人类 review 时间节省

### QA / Release Guardian

- 测试覆盖率
- 回归缺陷发现率
- 发布阻断准确率
- 发布后告警响应时间
- 回滚预案完整度

### Knowledge Steward

- 文档更新及时率
- 文档引用率
- 重复问题减少率
- 新人 onboarding 时间
- 过期文档识别率

## 4. 指标仪表盘建议

### 每日视图

- 今日 AI 完成任务数
- 今日 AI 创建 PR 数
- 待人类审批数
- 阻塞任务数
- 高风险操作请求数

### 每周视图

- AI PR 合并率
- 平均 Review 轮次
- AI 任务返工率
- 人类节省时间估算
- 主要错误模式 Top 5

### 每月视图

- 研发 Lead Time 变化
- 缺陷逃逸率变化
- AI 成本 / 交付价值
- 文档覆盖率变化
- 规则命中与失效情况

## 5. 质量回路

AI 研发系统需要持续学习：

```text
执行 → 评审 → 反馈 → 结构化 → 规则化 → 下一次执行
```

### 错误模式库

每类高频错误应被记录：

- 错误描述
- 发生位置
- 示例 PR
- 触发条件
- 修复建议
- 对应规则
- 是否已降低发生频率

### 规则有效性

每条规则应衡量：

- 命中次数
- 阻止的问题数
- 误报次数
- 是否需要更新
- 是否应废弃

## 6. 警戒指标

如果出现以下情况，应降低 AI 权限或重新训练规则：

- AI PR 缺陷逃逸率上升。
- AI 任务返工率连续上升。
- 人类 review 负担没有下降。
- AI 成本高于节省的人力价值。
- 高风险操作请求频繁出现。
- 同类错误重复发生。

## 7. 阶段性目标

第一阶段建议目标：

- 低风险任务从需求到 PR 周期缩短 30%-50%。
- 人类 review 时间下降 20%-40%。
- 文档和测试补齐率提高到 80% 以上。
- AI 生成 PR 的首次 CI 通过率稳定超过 70%。
- 高风险操作 100% 人类审批。
