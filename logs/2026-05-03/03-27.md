# 2026-05-03 03:27 — 状态机驱动的 AI 同事系统

## 本轮新增发现
AI 同事不是简单角色，而是带状态迁移的决策系统。能力不是关键，状态转移规则才是关键。研发效率的核心变量不是单点执行速度，而是任务在状态之间的等待时间。

## MOTX/Moxt 产品理解更新
Moxt 可进一步抽象为 Agent Runtime × State Machine Engine × Memory Graph。Rules 本质上是在定义状态迁移约束，而不是普通提示词。

## AI 同事角色设计
每个 AI 同事都应定义状态、转移条件和退出条件。AI Developer 可建模为 UNDERSTANDING → PLANNING → CODING → TESTING → BLOCKED → DONE；AI Reviewer、AI QA、AI DevOps 也应有各自子状态机。新增 AI Orchestrator，负责跟踪状态、识别阻塞并触发下一步动作。

## AI 原生工作空间设计
工作空间应支持 State Board、Blocked Queue、Decision Queue。人类不应处理所有任务，只处理 Decision Queue 中需要人工判断的问题。

## 软件研发新作业范式
任务 = 状态流 + Agent 推进。系统自动推进，遇到 BLOCKED 时分类阻塞原因并升级处理。

## 效率提升机制
等待时间压缩、状态可视化、自动推进、并行状态流和系统级优化。

## 风险与边界
状态爆炸、错误状态转移、Orchestrator 失效、阻塞误判、人类成为瓶颈。

## 阶段性结论
软件研发 AI 化的本质不是引入 AI，而是引入状态机驱动的工作系统。

## 下一轮研究方向
设计通用研发状态机、Blocked Protocol、Orchestrator 架构。
