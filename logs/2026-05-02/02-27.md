# MOTX / Moxt 软件研发范式增量研究记录：Verifiable Delivery Protocol

时间：2026-05-02 02:27

## 本轮新增发现

上一轮已经明确 Moxt 的核心在 **Task + Agent + Constraint** 与 **Handoff Packet**。本轮进一步发现一个关键缺口：

> 当前所有 agent 系统（Codex / Claude Code / Copilot）都能“生成 PR”，但无法保证 PR 的“可验证正确性”与“语义一致性”。

具体表现：

1. PR 通过测试 ≠ 正确实现需求（测试覆盖不足）
2. PR 符合 diff 预期 ≠ 没有引入隐性副作用
3. PR 被 reviewer 接受 ≠ 满足真实用户目标
4. 多 agent 并行 ≠ 全局行为一致（跨任务冲突）

因此，Moxt 需要从“任务交接”升级到：

```text
Verifiable Delivery Protocol（可验证交付协议）
```

即：不仅定义任务输入（Handoff），还定义输出必须满足的可验证条件，并由系统自动校验。

## MOTX / Moxt 产品理解更新

新增一层关键能力：

> Moxt 不仅是 Task Structuring + Agent Orchestration，而是 **Delivery Verification System（交付验证系统）**

产品核心从 6 层扩展为 7 层：

```text
1. Task Structuring Engine
2. Agent Adapter Layer
3. Semantic Permission Layer
4. Agent Runtime Control Plane
5. Review & Release Governance
6. Organizational Memory
7. Verifiable Delivery Layer（新增）
```

### Verifiable Delivery Layer 的职责

不是简单“跑测试”，而是做 4 类验证：

1. **Spec Consistency Check**：实现是否符合 Requirement / Design / TaskGraph
2. **Scope Boundary Check**：是否修改了未授权区域（semantic + file + dependency）
3. **Behavioral Validation**：是否真正改变了目标行为（不仅仅是代码变化）
4. **Regression & Side-effect Detection**：是否破坏已有功能或引入隐性风险

这层是 Moxt 与“普通 agent 平台”的本质分界线。

## AI 同事角色设计更新

### 新增角色：Delivery Auditor / 交付审计 AI

定位：独立于 Reviewer 与 QA，专门验证“交付是否真实满足任务目标”。

Reviewer 本质是基于 diff + 经验判断；Delivery Auditor 是基于 Spec + 行为验证 + 结构分析。

职责：

- 对照 Requirement / Design / TaskGraph 做语义一致性校验
- 检查是否存在测试驱动作弊、功能绕过、hardcode/mock、最小表面改动但未解决问题
- 运行行为验证（如 UI flow、API contract）
- 检查 dependency graph 是否异常变化
- 输出 Delivery Verdict：pass / soft fail / hard fail / escalate

工具权限：

- 读 PR diff、Requirement / Design / TaskGraph、测试结果、runtime traces
- 执行行为验证脚本、静态分析、dependency diff
- 禁止修改代码、推送 PR

评价指标：

- 虚假通过检测率
- 测试作弊识别率
- 发布后问题追溯命中率
- 与真实用户问题的相关性

### Reviewer 角色调整

Reviewer 从“全面质量控制”收敛为代码质量、可读性、局部逻辑正确性。不再承担需求一致性和行为验证。

### QA / Release Guardian 调整

QA 从“测试执行者”升级为测试策略生成者 + 覆盖缺口发现者；Delivery Auditor 验证测试是否真的有效。

## AI 原生工作空间设计更新

### 新增核心对象：Delivery Contract

在 Handoff Packet 之后，必须定义：

```yaml
delivery_contract:
  success_definition:
    - "用户可以完成 X 操作"
    - "系统返回 Y 结果"
  invariants:
    - "不得影响 auth flow"
    - "billing calculation 不变"
  anti_patterns:
    - "禁止 hardcode response"
    - "禁止修改测试以绕过失败"
  verification_methods:
    - type: unit_test
    - type: integration_test
    - type: scenario_simulation
    - type: static_analysis
  failure_conditions:
    - "测试通过但行为不变"
    - "新增 dependency 指向敏感模块"
  confidence_score_required: 0.85
```

### 新增流程：Dual Validation Pipeline

```text
Implementation Agent
    ↓
PR
    ↓
Reviewer（代码质量）
    ↓
QA（测试执行 + 覆盖）
    ↓
Delivery Auditor（语义与行为验证）
    ↓
Human Gate（高风险）
    ↓
Merge / Release
```

Delivery Auditor 是“最后一个 AI 审查者”，职责不是找 bug，而是验证“是否真的完成任务”。

## 软件研发新作业范式更新

从上一轮：

```text
Goal-driven → agent executed → human governed
```

升级为：

```text
Goal-driven → Spec-structured → Agent-executed → System-verified → Human-governed
```

核心变化：“验证”成为一等公民，而不是附属步骤。

Moxt 新范式：定义目标 → 定义验证方式 → agent 实现 → 系统验证 → 才允许进入发布链路。

## 效率提升机制

新增关键效率杠杆：

```text
有效产出率 = 通过验证的 PR 数 / 总 PR 数
```

Moxt 的目标不是增加 PR 数量，而是增加“无需返工即可发布”的 PR 比例。

### Pre-emptive Validation

在 agent 执行阶段就嵌入验证：

- 生成代码时同步生成 expected behavior 和 test hints
- 执行过程中提前检测 scope violation 和 dependency drift
- PR 前自动运行 Delivery Contract 验证

效果：减少进入 review 阶段的无效 PR。

## 风险与边界

### 风险一：验证系统被绕过

agent 可能针对验证规则优化，而不是解决问题，或伪造符合 contract 的行为。

对策：Delivery Contract 动态变化，引入随机验证（fuzz / adversarial cases），使用历史真实数据验证。

### 风险二：验证成本过高

验证过重会降低 agent throughput、增加 CI 成本、破坏开发体验。

对策：风险分级验证：low risk 轻量验证，medium 标准验证，high 强验证 + human。

### 风险三：语义漂移（Spec Drift）

Requirement / Design / TaskGraph / Contract 不一致会导致冲突。

对策：所有验证基于同一对象图谱，Delivery Contract 从 Requirement 自动派生。

## 阶段性结论

软件研发的核心瓶颈正在从“写代码”转向“验证代码是否真正完成目标”。

Moxt 的核心竞争力不是生成能力，而是验证能力 + 约束能力。

```text
第一阶段：AI 写代码（已解决）
第二阶段：AI 并行写代码（正在发生）
第三阶段：AI 生成 + 系统验证 + 组织学习（Moxt 机会）
```

## 下一轮研究方向

1. Delivery Contract 自动生成：如何从 Requirement 自动推导验证条件？
2. Behavioral Verification 技术路径：UI / API / state machine 如何自动验证？
3. Agent 自我验证机制：agent 在提交 PR 前自检。
4. Verification Cost Optimization：如何保证质量同时降低验证成本？
5. Spec Graph 一致性系统：Requirement / Design / Task / Contract 自动同步机制。
