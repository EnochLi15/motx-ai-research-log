# Agent Sandbox 与执行隔离层（Execution Isolation Layer）

时间：2026-05-02 18:27

## 本轮新增发现

在上一轮我们明确了 **Agent Identity + PermissionGrant + DelegationChain**，本轮继续往“可落地控制”推进，关键缺口是：

> **即使身份与权限正确，如果执行环境不可控，系统仍然不安全。**

当前主流 coding agent 有一个共性：能生成代码，能提交 PR，但执行环境边界模糊。

例如：

```text
- 是否允许访问外网？
- 是否允许读取 secrets？
- 是否允许执行 shell？
- 是否允许运行 migration？
- 是否允许访问生产资源？
```

这些问题如果只靠 PermissionGrant 是不够的，因为 Permission 是“允许做什么”，但没有约束“在哪里做、怎么做”。

因此，本轮提出一个关键新增层：

> **Agent Sandbox / Execution Isolation Layer（执行隔离层）**

这将成为 Moxt 真正区别于 GitHub / IDE / Agent 框架的核心能力之一。

## MOTX/Moxt 产品理解更新

当前模型升级为三层控制体系：

```text
Layer 1: Identity & Permission
- AgentIdentity
- PermissionGrant
- DelegationChain

Layer 2: Execution Control
- Agent Sandbox
- Tool Proxy
- Network Boundary
- Data Access Boundary

Layer 3: Verification & Acceptance
- EvidencePack
- RiskProfile
- ReleaseDecision
```

关键变化：

> Moxt 不只是“谁可以做什么”，而是“谁在什么隔离环境下做什么，并产生什么证据”。

新的产品定义可以升级为：

> **Moxt = Agentic Engineering Control Plane with Identity + Isolation + Evidence**

## AI 同事角色设计更新

新增一个关键角色（系统级 AI）：

### Sandbox Orchestrator / 执行隔离调度器

**定位**：为每个 agent 执行任务创建、配置、监控、销毁隔离执行环境。

### 职责

- 为每个任务实例创建 sandbox（container / VM / ephemeral env）；
- 根据 PermissionGrant 配置 sandbox：文件系统范围、网络访问策略、secret 注入策略、tool proxy；
- 控制 agent 的 runtime 能力：是否允许 shell、是否允许 package install、是否允许 test execution；
- 记录 execution trace；
- 在异常时 kill sandbox；
- 在任务完成后回收资源；
- 向 EvidenceBuilder 输出执行证据。

### 工具权限

- 可调用 container runtime、network policy engine、secret manager（只可注入，不可读取明文）；
- 不可修改 repo 主分支；
- 不可 approve PR；
- 不可修改业务逻辑。

### 协作边界

- 不参与代码质量判断（交给 Reviewer / EvidenceBuilder）；
- 不参与发布决策（交给 Release Governor）；
- 不决定 agent 是否可信（交给 Agent Identity Steward）。

### 评价指标

```text
sandbox isolation breach rate
unexpected network call rate
secret exposure incidents
sandbox startup latency
execution reproducibility rate
ghost sandbox rate
```

## 软件研发新作业范式更新

范式进一步升级：

> **Isolated + Identity-Bound + Evidence-Gated Agentic Development**

核心流程：

```text
1. IntentSpec
2. ChangePlan
3. Agent Identity Check
4. PermissionGrant
5. Sandbox Provisioning
6. Agent Execution (in sandbox)
7. ExecutionTrace
8. EvidencePack
9. RiskProfile
10. ReleaseDecision
```

关键变化：

1. 所有 agent 执行必须在 sandbox 内。不允许 agent 直接操作 developer 本地环境、直接访问生产资源、直接调用外部工具。
2. ExecutionTrace 成为一等公民。以前只看 diff + test，现在看 diff + test + execution trace。
3. Tool 调用必须经过 Proxy，防止 agent 私自调用 npm publish、cloud deploy、db migration 等高风险工具。

## 效率提升机制

这一层不是降低成本，而是**解锁更高自动化比例**。

### 1. 安全自动执行更多任务

有 sandbox 后，可以放心自动执行 test generation、refactor、dependency update、code formatting、bug reproduction。

### 2. 提高执行并行度

因为每个 agent 在独立 sandbox 中运行，可以 N 个 agent 并行尝试 solution。

### 3. 提高结果可复现性

sandbox + trace 可以做到：相同输入、相同环境、相同输出。这对调试 agent failure 极其关键。

### 4. 降低 infra 风险成本

避免 agent 误操作真实资源、触发 CI/CD、修改生产配置。

## 风险与边界

### 1. Sandbox 并不等于安全

潜在问题包括 side-channel、依赖污染、sandbox escape、supply chain 注入。

### 2. 成本爆炸风险

每个任务一个 sandbox 会导致 CPU / memory / storage 成本激增，需要 TTL、lazy execution、result caching。

### 3. Tool Proxy 复杂度

需要定义每个工具的安全模型、参数 masking、调用策略、审计规则。

### 4. Developer friction

如果设计不好，sandbox 变慢会导致开发体验下降。需要 warm pool、incremental environment、cache layer。

## 阶段性结论

Moxt 的控制面已经形成完整闭环：

```text
Identity（谁）
→ Permission（能做什么）
→ Sandbox（在哪里做）
→ ExecutionTrace（做了什么）
→ Evidence（证明什么）
→ Decision（是否接受）
```

核心洞察升级为：

> 软件研发的 AI 问题，本质不是“生成代码”，而是“在受控环境中执行变更，并证明其可被接受”。

最终产品定位进一步收敛为：

> **Moxt = AI Software Change Execution & Acceptance Control Plane**

## 下一轮研究方向

1. 设计 Sandbox Runtime 架构选择：container vs microVM vs WASM，不同风险等级映射。
2. 设计 Tool Proxy DSL：如何声明 tool 权限，如何做参数级控制。
3. 设计 ExecutionTrace → EvidencePack 映射规则。
4. 设计 Sandbox 成本优化策略：warm pool、shared base image、diff execution。
5. 设计 Agent 多候选并行执行策略：N agent → N sandbox → SolutionSet。
6. 设计高风险操作隔离策略：migration sandbox、infra sandbox、secret-free execution。
7. 设计本地开发环境与 Moxt sandbox 的融合模式：dev → agent → sandbox → PR → review。
