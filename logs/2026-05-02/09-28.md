# MOTX / Moxt 软件研发范式研究增量更新：TaskGraph Runtime 与可控搜索验证系统

时间：2026-05-02 09:28

## 本轮新增发现

上一轮已经明确 Moxt 是 control plane。本轮关键突破在于：

> **TaskGraph 必须从“静态拆解结构”升级为“动态调度系统”。**

当前外部 agent（Codex / Copilot / Claude Code）已经具备长时间运行能力、多任务并行、自动 PR 生成、环境隔离执行，但它们没有一个统一的“任务调度与依赖控制模型”。

### 新增关键洞察

TaskGraph 如果只是：

```text
Task A → Task B → Task C
```

那它仍然只是 Jira。

真正需要的是：

```text
TaskGraph = DAG + Runtime State + Budget + Risk + Agent Binding
```

每个 Task 节点必须具备：

```yaml
task:
  id: TASK-123
  type: implementation | test | migration | refactor
  dependencies:
    - TASK-101
  execution:
    agent_candidates:
      - codex
      - claude_code
    parallelizable: true
  runtime:
    status: pending | running | blocked | failed | success
    retries: 0
    assigned_agent: codex
  constraints:
    max_diff_files: 8
    max_runtime_minutes: 30
  risk:
    level: medium
    domains:
      - billing_ui
```

结论：TaskGraph 是 Moxt 的核心“调度引擎接口”，不是文档结构。

## MOTX / Moxt 产品理解更新

本轮把 Moxt 再推进一层抽象：

```text
Moxt =
TaskGraph Runtime
+ Agent Orchestration
+ Delivery Verification
+ Risk Governance
```

不是：

```text
AI IDE
Chat Workspace
Prompt Layer
```

### 新增关键区分

```text
Coding Agent（Codex / Copilot）解决：
→ 如何写代码

Moxt 解决：
→ 写什么代码
→ 谁来写
→ 写到什么程度
→ 是否可以合并

→ 是否值得这个成本
```

## AI 同事角色设计更新

本轮不增加角色数量，而是强化“角色之间的协作协议”。

### Role Contract（角色契约）

每个 AI 同事不只是职责，而是必须定义：

```yaml
role_contract:
  inputs:
  outputs:
  cannot_do:
  escalation_rules:
  success_metrics:
```

### Task Architect 重大升级

上一轮只是“拆任务”，本轮升级为：

```text
Task Architect = TaskGraph Compiler
```

新职责：

1. 把 Requirement 编译为 TaskGraph DAG。
2. 标注任务并行性（parallelizable）。
3. 标注风险域（semantic risk）。
4. 标注执行策略（agent candidates）。
5. 插入验证节点（test / audit / review）。
6. 控制粒度（避免过大或过碎）。

新输出：

```yaml
task_graph:
  nodes:
    - TASK-1 (design validation)
    - TASK-2 (implementation)
    - TASK-3 (test generation)
    - TASK-4 (delivery audit)
  edges:
    - TASK-1 → TASK-2
    - TASK-2 → TASK-3
    - TASK-3 → TASK-4
```

Moxt 的“任务质量”核心由 Task Architect 决定，而不是 Implementation Engineer。

### Coordinator 升级为执行控制器

新增能力：

- 动态分配 agent backend。
- 控制并发，避免 PR flood。
- 处理任务失败重试。
- 按预算中断任务。
- 检测 deadlock（任务互相等待）。

新增关键指标：Task Throughput、Blocked Task Ratio、Retry Efficiency、Parallel Execution Gain。

### Delivery Auditor 强化

新增职责：不仅验证“完成”，还要验证是否最小改动完成、是否违反架构边界、是否存在“测试欺骗”。

新增能力：diff vs requirement alignment、invariant check、anti-overfitting detection。

### Security Sentinel 细化

本轮新增：

```text
ActionIntent → Risk Escalation Matrix
```

例如：

```text
modify billing logic → critical
change auth middleware → critical
edit test only → suspicious
change CI workflow → high
```

## AI 原生工作空间设计

本轮新增一个关键层：**Execution Timeline（执行时间轴）**。

### 为什么需要

当多个 agent 并行时，最大问题不是“谁在干什么”，而是：

```text
什么时候干了什么
为什么做这个决定
谁触发了风险
```

### Execution Timeline 示例

```text
09:00 TASK-1 started (codex)
09:02 TASK-1 completed
09:03 TASK-2 assigned (claude_code)
09:10 TASK-2 modified billing.ts ⚠️ high risk
09:11 Security Sentinel triggered
09:12 Human approval requested
09:20 approved
09:25 TASK-2 completed
09:27 TASK-3 started
```

这是 Moxt 的“可审计性核心 UI”。

## 软件研发新作业范式

本轮新增关键转变：

```text
从 Task Pipeline
→ Search + Selection + Verification Loop
```

### 新范式强化版

```text
Goal
→ TaskGraph（定义搜索空间）
→ 多 Agent 并行尝试（solution candidates）
→ Delivery Auditor 评估多个候选
→ 选择最优解（不是第一个解）
→ Human Gate（关键决策）
→ Memory Update
```

关键变化：

```text
传统：一个人写 → 一个 PR → 合并
Moxt：多个 agent 并行写 → 多个候选 PR → 系统选最优 → 人类确认
```

研发从“单路径执行”变成“多路径搜索问题”。

## 效率提升机制

### Candidate Competition（候选竞争机制）

```text
同一个 Task → 多个 agent 同时执行 → 选择最优 PR
```

适用场景：复杂逻辑、性能优化、架构设计分歧。收益是避免锁死在错误实现路径。

### Early Kill（早停机制）

Coordinator 应支持：

```text
if progress < threshold after N minutes:
    terminate agent
```

避免长时间低质量执行。

### Verification First（验证前置）

在 TaskGraph 中：

```text
Test / Audit 节点必须在 Implementation 之前定义
```

避免写完才发现标准不清。

### Budget-aware Execution

每个任务必须有：

```yaml
budget:
  max_cost
  max_runtime
  max_attempts
```

否则系统会无限尝试导致成本爆炸。

## 风险与边界

### Search Explosion（搜索爆炸）

引入多 agent + 多候选后：

```text
任务数 × agent 数 × retry 数 → 指数级增长
```

解决方案：candidate limit、parallelism cap、budget control、early pruning。

### False Positive Completion

表现：测试全绿，但用户目标未完成。

原因：测试设计错误、agent 修改测试、mock 过度。

解决：Delivery Auditor + Scenario Simulation。

### Architecture Drift

多个 agent 并行导致模块耦合增加、重复逻辑、风格不一致。

解决：Architecture Governor。

## 阶段性结论

本轮最重要的结论：

```text
Moxt 的核心不是“让 AI 写代码更快”，
而是“把软件研发转化为一个可控的搜索与验证系统”。
```

进一步抽象：

```text
软件研发 = 在约束条件下搜索最优实现
```

Moxt 的职责：

```text
定义搜索空间（TaskGraph）
限制搜索风险（Permission / Budget）
执行搜索（Agent）
验证结果（Delivery Auditor）
选择最优（Coordinator + Human）
沉淀经验（Memory）
```
