# MOTX / Moxt 软件研发范式增量研究记录：Interpretability & Governance Layer

时间：2026-05-02 07:28

## 本轮新增发现

在引入 Incentive Layer 后，系统已经具备：

```text
执行（Execution）
验证（Verification）
收敛（Convergence）
策略（Strategy）
资源（Economic）
对齐（Incentive）
```

但本轮识别出一个新的系统性瓶颈：

> **当系统具备复杂多层决策能力后，“为什么做出某个决策”变得不可理解，从而导致人类无法信任、无法干预、无法审计。**

具体问题：

1. Task 被修改多次后，人类无法理解最终 Task 为什么这样定义。
2. Agent 选择路径（为何选 Codex 而不是 Claude）不可解释。
3. Strategy Layer 的 policy 调整缺乏透明性。
4. Incentive Layer 调整后行为变化无法归因。
5. 出现线上问题时，无法追溯：是 Task 问题、策略问题，还是激励问题。

因此系统需要补上关键能力：

> **Interpretability & Governance（可解释性与治理）**

## MOTX / Moxt 产品理解更新

在 11 层基础上新增第 12 层：

```text
12. Interpretability & Governance Layer（可解释性与治理层）
```

### 核心作用

不是简单 logging，而是：

```text
让系统每一个关键决策都：
- 可解释（explainable）
- 可追溯（traceable）
- 可审计（auditable）
- 可干预（controllable）
```

### 三类必须解释的决策

#### 1. Structural Decisions（结构决策）

- 为什么这样拆 Task？
- 为什么修改 Task？

#### 2. Execution Decisions（执行决策）

- 为什么选择某个 agent？
- 为什么增加 budget？

#### 3. Policy Decisions（策略决策）

- 为什么提高 verification level？
- 为什么限制某模块？

## AI 同事角色设计更新

### 新增角色：Decision Explainer / 决策解释 AI

#### 定位

将系统内部复杂决策转化为“人类可理解解释”的专职角色。

#### 核心职责

对每个关键动作生成 Explanation：

```yaml
decision_explanation:
  decision: "increase verification level"
  reason:
    - "billing module failure rate increased to 0.42"
    - "recent 3 tasks failed due to regression"
  alternatives_considered:
    - "keep current level (rejected)"
  expected_impact:
    - "reduce regression risk"
    - "increase cost by ~15%"
```

#### 工具权限

- 读所有 system logs、Task Evolution、Strategy decisions。
- 写 explanation records（不可修改决策）。

#### 评价指标

- 人类理解度（可读性评分）。
- 解释覆盖率（关键决策是否都有解释）。
- 解释准确性（是否真实反映原因）。

### 新增角色：Governance Auditor / 治理审计 AI

#### 定位

检查系统行为是否符合组织规则、合规要求与内部策略。

#### 职责

- 审计是否绕过权限系统、是否违反 policy、是否存在异常决策路径。
- 检测 shadow workflow（绕过标准流程）与 unjustified decision（无解释决策）。
- 输出 Governance Report。

示例：

```text
Governance Report：
- violation_detected: true
- severity: high
- affected_modules: ["billing"]
```

#### 工具权限

- 读全部日志与决策记录。
- 可阻断高风险流程。
- 不直接执行任务。

#### 评价指标

- 未解释决策比例下降。
- 违规行为检测率。
- 审计误报率。
- 合规事件减少。

## AI 原生工作空间设计更新

### 新增核心对象：Decision Graph（决策图谱）

这是本轮最关键结构升级。

#### 定义

```text
所有 Task、Agent、Policy、Budget、Incentive 的决策关系图
```

#### 示例

```yaml
decision_graph:
  nodes:
    - task_v3
    - agent_selection_codex
    - verification_upgrade
    - policy_update_billing
  edges:
    - from: task_v2
      to: task_v3
      reason: "refined acceptance criteria"
    - from: billing_failures
      to: verification_upgrade
    - from: verification_upgrade
      to: agent_selection_codex
```

### 新增机制：Decision Replay（决策回放）

允许回放某个 PR / Bug 的完整决策路径。

用途：

- Debug 系统行为。
- 审计。
- 培训人类团队。

### 新增机制：Human Override Interface

允许人类覆盖：

```text
- agent selection
- task definition
- policy constraint
```

并记录 override_reason，用于系统学习。

## 软件研发新作业范式更新

从上一轮：

```text
自执行 + 自优化 + 自对齐
```

升级为：

```text
自执行
+ 自优化
+ 自对齐
+ 可解释
+ 可治理
```

### 新范式核心特征

1. 系统不仅“做决策”，还“解释决策”。
2. 人类从“执行者”转为“监督者 + 干预者”。
3. 每个结果都可以追溯到原因链条。

## 效率提升机制

新增关键指标：

```text
Debug Efficiency（调试效率）= 问题定位时间 / 总问题数
```

### Failure Traceability

当出现问题时，直接定位：Task 问题、Agent 问题、Strategy 问题、Incentive 问题，而不是人工排查。

### Explainable Optimization

优化不仅发生，还必须解释为什么优化有效，使系统可以被信任、被调整、被扩展。

## 风险与边界

### 风险一：解释成本过高

生成解释本身消耗资源：token 成本、latency 增加。

对策：只对关键决策生成完整解释。

### 风险二：伪解释（Post-hoc Rationalization）

系统可能编造“看起来合理”的解释。

对策：explanation 必须基于真实 decision log，禁止生成未发生的原因。

### 风险三：过度治理导致系统僵化

Governance 过强会限制 agent 自由度、降低效率。

对策：区分 critical governance（必须）与 soft governance（建议）。

## 阶段性结论

当 AI 研发系统复杂度达到一定程度后，“可解释性”与“治理能力”成为核心基础设施，而不是附加功能。

完整系统模型演进为：

```text
Execution
+ Verification
+ Convergence
+ Strategy
+ Economic
+ Incentive
+ Interpretability & Governance
```

进一步抽象：

```text
Moxt = 可执行 + 可收敛 + 可优化 + 可对齐 + 可解释 的工程系统
```

或者：

```text
Moxt 是一个“可被信任的 AI 研发组织操作系统”。
```
