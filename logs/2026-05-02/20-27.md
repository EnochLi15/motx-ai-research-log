# Decision Engine 与可解释发布决策系统（Explainable Release Decision System）

时间：2026-05-02 20:27

## 本轮新增发现

在前几轮中已经建立：

```text
Identity → Permission → Sandbox → Execution → Verification Pipeline → EvidencePack
```

但仍存在一个关键未闭合环节：

> **ReleaseDecision 仍然是“黑箱策略”或简单规则，而不是一个可解释、可学习、可优化的决策系统。**

当前问题：

```text
- 为什么这个 PR 被自动 merge？
- 为什么这个 PR 被拒绝？
- 为什么需要人工 review？
- 不同 agent 的结果如何比较？
```

如果没有结构化决策层，会导致：无法建立用户信任，无法优化系统策略，无法做 agent 之间的选择，无法做长期学习。

## MOTX/Moxt 产品理解更新

系统从“验证控制”进一步升级为：

```text
Moxt = Decision-Centric AI Engineering Control System
```

即：

> **核心不只是验证，而是“如何基于证据做决策”。**

系统结构升级为五层：

```text
Layer 1: Identity & Permission
Layer 2: Sandbox (Execution)
Layer 3: Verification Pipeline
Layer 4: Evidence Modeling
Layer 5: Decision Engine
```

## AI 同事角色设计更新

新增核心角色：

### Decision Engine / 发布决策引擎（系统级 AI）

**定位**：基于 EvidencePack、RiskProfile、AgentPerformance，做出是否接受变更的决策，并输出可解释原因。

### 职责

- 聚合输入：EvidencePack、VerificationResult、RiskProfile、AgentPerformanceRecord、DelegationChain；
- 计算 acceptance score 与 risk-adjusted score；
- 输出 decision（accept / reject / require human / request changes）；
- 输出 explanation（结构化原因）和 confidence level；
- 支持多候选方案选择（SolutionSet）；
- 支持策略演化（learning）。

### 工具权限

- 可读取所有控制面对象；
- 不可直接修改代码；
- 不可绕过验证流程；
- 可触发 re-evaluation；
- 可请求 additional verification。

### 协作边界

- 不生成代码（Worker）；
- 不设计验证（Verification Architect）；
- 不控制权限（Identity Steward）；
- 专注于“是否接受 + 为什么”。

### 评价指标

```text
decision accuracy（事后正确率）
false accept rate（错误放行）
false reject rate（误拒）
human override rate
explanation usefulness score
multi-agent selection quality
```

## 软件研发新作业范式更新

范式升级为：

> **Decision-Driven Agentic Development**

核心思想：

```text
不是“生成代码 → 验证 → merge”
而是“生成候选解 → 评估 → 选择最优解”
```

流程：

```text
1. IntentSpec
2. ChangePlan
3. Delegation → 多 agent 并行执行
4. Candidate SolutionSet（多个 PR）
5. Verification Pipeline（每个候选）
6. EvidencePack（每个候选）
7. Decision Engine（选择 or 拒绝）
8. ReleaseDecision
```

### 核心变化

#### 1. 单路径 → 多候选搜索

```text
过去：1 个 agent → 1 个 PR
现在：N 个 agent → N 个 PR → 选择最优
```

#### 2. Merge 变为“选择问题”

不是“是否 merge”，而是“选择哪个方案”。

#### 3. Decision 可解释

系统必须回答：为什么选 A 而不是 B？

## 效率提升机制

### 1. 多 agent 并行 → 最优解提升

不同 agent 具有不同策略、不同风格和不同质量，Decision Engine 选择最优。

### 2. 自动化决策替代人工 review

在低风险场景下，Decision Engine 可以承担 Senior Reviewer 的部分判断职责。

### 3. 减少无效 PR

自动 reject 低质量、高风险、低信心结果。

### 4. 持续优化决策策略

通过：

```text
Decision → Outcome → Feedback → Policy update
```

持续优化。

## 风险与边界

### 1. 决策错误风险

错误接受最危险，错误拒绝会影响效率。

### 2. 模型偏见

Decision Engine 可能偏好某些 agent、某些代码风格或某些低成本路径。

### 3. 可解释性不足

如果 explanation 不可信，系统不可被企业接受。

### 4. 多候选成本

N 个 agent 可能导致 N 倍成本，需要 early stop、pruning、heuristic selection。

## 阶段性结论

Moxt 的完整系统已经形成：

```text
Intent
→ Plan
→ Identity
→ Permission
→ Sandbox
→ Execution
→ Verification
→ Evidence
→ Decision
```

最终核心洞察：

> 软件研发的本质不是代码生成，也不是测试，而是“在不确定性中做正确决策”。

因此：

> **Moxt 的核心不是 coding system，而是 decision system。**
