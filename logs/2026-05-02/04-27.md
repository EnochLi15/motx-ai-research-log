# MOTX / Moxt 软件研发范式增量研究记录：Strategy Layer 与全局优化

时间：2026-05-02 04:27

## 本轮新增发现

上一轮已经把系统推进到 **Task Evolution（任务进化）**，解决“任务本身可能是错的”问题。本轮进一步发现一个更高阶问题：

> 即使每个 Task 都在收敛，整个系统仍可能收敛到“局部最优”而非“全局最优”。

具体表现：

1. 多个 Task 分别成功，但组合后系统复杂度过高（架构退化）。
2. agent 倾向选择“最容易实现”的方案，而非“长期最优”的方案。
3. Task Evolution 只在局部调整，缺乏跨任务、跨模块的全局优化。
4. 技术债以“成功 PR”的形式被持续引入（隐性积累）。

因此，系统需要从：

```text
Task-level convergence
```

升级为：

```text
System-level optimization
```

引出本轮核心新增概念：

> **Strategy Layer（策略层）**：负责跨任务、跨周期的全局最优控制。

## MOTX / Moxt 产品理解更新

在已有 8 层架构基础上，新增第 9 层：

```text
9. Strategy & Global Optimization Layer（策略与全局优化层）
```

### Strategy Layer 核心职责

不是直接参与单个任务，而是：

```text
观察整个研发系统 → 识别长期趋势 → 调整策略
```

包括：

1. **架构健康度控制**
   - 检测模块耦合上升
   - 检测重复实现
   - 检测技术债积累

2. **任务策略调整**
   - 是否应继续拆分？
   - 是否应合并任务？
   - 是否应暂停某类任务？

3. **agent 使用策略**
   - 哪类任务适合哪个 agent
   - 哪些 agent 在某些模块表现差

4. **探索 vs 收敛平衡**
   - 是否需要尝试不同实现路径？
   - 是否过早收敛？

### 系统升级路径

```text
阶段 1：写代码（单 agent）
阶段 2：并行执行（multi-agent）
阶段 3：任务验证（verification）
阶段 4：任务进化（task evolution）
阶段 5：策略优化（strategy layer）
```

## AI 同事角色设计更新

### 新增角色：Strategy Planner / 策略规划 AI

#### 定位

站在“研发组织层面”，而不是“任务层面”的 AI 同事。

#### 核心职责

- 分析全部 Task Evolution Logs、PR 成功率 / 返工率、模块级失败分布。
- 识别模式：某模块持续失败需要重构；某类任务成功率低需要调整拆分策略；agent 在某类任务表现差需要替换或限制。
- 输出 Strategy Recommendations：refactor module X、stop auto-execution on Y tasks、increase verification level for Z domain、merge tasks A/B/C into single design change。

#### 工具权限

- 读全部 TaskGraph、Task Evolution Logs、PR / review / auditor 数据。
- 写 strategy proposals、system-level policy updates（需审批）。
- 禁止直接修改代码、直接触发执行 agent。

#### 评价指标

- 长期返工率下降。
- 技术债增长速度下降。
- 系统复杂度稳定性。
- agent 使用效率提升。

### 新增角色：Architecture Governor / 架构治理 AI

#### 定位

Strategy Planner 的执行者，负责“架构层约束落地”。

#### 职责

- 定义模块边界（module boundaries）。
- 限制跨模块调用。
- 标记“需要重构”的区域。
- 在 Task Architect 阶段插入约束。

示例：

```yaml
architecture_constraints:
  forbidden_dependencies:
    - billing → auth direct call
  required_patterns:
    - use service layer
```

#### 工具权限

- 可修改架构规则、lint / static analysis 规则。
- 可阻止违反架构的 PR。
- 不直接改业务逻辑。

#### 评价指标

- 架构违规 PR 数量。
- 模块耦合度变化。
- 重构成功率。
- 技术债减少趋势。

## AI 原生工作空间设计更新

### 新增核心对象：Strategy Snapshot

周期性记录系统状态：

```yaml
strategy_snapshot:
  timestamp: 2026-05-02
  system_metrics:
    task_success_rate: 0.72
    avg_convergence_steps: 2.8
    pr_rework_rate: 0.34
  module_health:
    billing:
      risk: high
      repeated_failures: 5
    auth:
      stable: true
  agent_performance:
    codex:
      strong: ["api tasks"]
      weak: ["refactor"]
    claude_code:
      strong: ["large diff"]
  recommendations:
    - refactor billing module
    - restrict auto execution for migration tasks
```

### 新增机制：Policy Injection

Strategy Layer 不直接干预任务，而是通过：

```text
Policy → 注入到 Task Architect / Security / Auditor
```

例如：

```yaml
policy_update:
  domain: billing
  change:
    auto_execution: disabled
    required_human_review: true
    verification_level: high
```

## 软件研发新作业范式更新

从上一轮：

```text
Goal → Execute → Verify → Evolve → Converge
```

升级为：

```text
Goal
 → Spec
 → Execute
 → Verify
 → Evolve Task
 → Converge Locally
 → Strategy Layer 调整
 → 全局再优化
```

### 新范式核心特征

1. 局部收敛（Task） + 全局优化（Strategy）。
2. 系统持续自我调参（self-tuning）。
3. 架构不再是一次性设计，而是动态演化。

## 效率提升机制

新增关键指标：

```text
Global Efficiency（全局效率）= 有效交付价值 / 总工程成本
```

### 新机制：Anti-Local-Optimization

防止系统陷入“局部最优”：

#### 1. Cross-Task Analysis

- 多个 Task 修改同一模块 → 提示合并。
- 多个 PR 实现类似逻辑 → 提示抽象。

#### 2. Refactor Trigger

当出现 repeated failures、high patch density、increasing complexity，自动触发 Refactor Task，而非继续 patch。

### 新机制：Agent Portfolio Optimization

动态调整 agent 组合。例如：Codex 处理高效小任务，Claude Code 处理复杂重构，人类处理高风险设计。

## 风险与边界

### 风险一：策略层过度干预

可能导致系统变慢、agent 自主性下降、创新减少。

对策：策略层只提供“约束与建议”，不直接控制执行。

### 风险二：全局指标误导

例如优化 PR 成功率可能牺牲创新，优化稳定性可能降低开发速度。

对策：多目标优化：速度、质量、可维护性，不设单一 KPI。

### 风险三：架构僵化

Architecture Governor 过强会导致系统难以演化、新模式难引入。

对策：引入“实验区”（experimental zone），允许部分模块不受严格约束。

## 阶段性结论

软件研发 AI 系统必须同时具备三层能力：

```text
1. 执行能力（agent）
2. 收敛能力（task evolution）
3. 全局优化能力（strategy layer）
```

进一步抽象：

```text
Moxt = 多 agent 执行系统 + 问题收敛系统 + 策略优化系统
```

或者更本质地说：

```text
Moxt 是一个“持续逼近最优解”的工程系统。
```

## 下一轮研究方向

1. Strategy Decision Model：如何量化“是否需要重构”，如何避免误触发？
2. Architecture Health Metrics：如何自动计算模块复杂度与耦合度？
3. Policy Injection 机制设计：如何影响 Task 而不破坏灵活性？
4. Exploration Budget 分配：系统如何决定何时尝试新方案？
5. Agent Portfolio Learning：如何让系统自动学习“哪个 agent 在什么任务最优”？
