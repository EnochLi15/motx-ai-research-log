# Moxt 进入多 Agent 调度与解空间搜索阶段

时间：2026-05-02 13:27

## 一、本轮新增发现

### 1. 关键跃迁：系统瓶颈从控制转向调度策略

前两轮我们已建立：控制（Control Contract / Risk / Governance）与角色（Controller-based AI teammates）。本轮新增关键判断：

> 当 AI 同事体系建立后，系统瓶颈不再是“能不能控制 AI”，而是“如何调度 AI 才最优”。

| 阶段 | 核心问题 |
|---|---|
| 阶段 1 | AI 能不能用 |
| 阶段 2 | AI 能不能控 |
| 阶段 3 | AI 如何调度最优 |

### 2. 多 AI 协作本质不是协作，而是竞争 + 选择

新增关键模型：

> 协作只是表象，底层是：多候选解生成 + 选择机制。

也就是：

```text
不是：A agent + B agent 一起做
而是：A/B/C agent 各自做 → 系统选择最优
```

这与传统工程完全不同：

| 传统 | Moxt |
|---|---|
| 单实现 → review | 多实现 → 选择 |
| 人类判断 | 系统评分 + 人类介入 |

### 3. 引入核心新对象：Solution Set（解空间）

之前只有 Task → PR。本轮新增：

```text
Task → Solution Set → Selected Solution
```

Solution Set 包含：多个候选实现、多个测试方案、多个架构路径。

这是 AI 时代研发的本质变化：不再追求一次正确，而是生成解空间再收敛。

### 4. 调度策略成为一等产品能力

Moxt 的潜在核心能力不是 prompt 或 agent，而是 Scheduling Strategy（调度策略）。

包括：什么时候并行、什么时候只用一个 agent、什么时候 early stop、什么时候 escalate 给人。

## 二、Moxt 产品理解更新

### 从 Control Plane 到 Adaptive Orchestration System

升级定义：

> Moxt = Adaptive Multi-Agent Orchestration System。

新增关键词：

1. **Adaptive（自适应）**：根据任务类型调整策略，根据历史表现调整 agent 选择。
2. **Orchestration（编排）**：不只是调度，而是管理执行顺序、依赖、并发。
3. **Multi-Agent Optimization（多代理优化）**：目标不是执行任务，而是找到最优解。

### 新的六层模型

```text
1. Intent Layer（人类意图）
2. Constraint Layer（控制契约）
3. Orchestration Layer（新增核心）
4. Execution Layer（agents）
5. Verification & Selection Layer
6. Memory & Learning Layer
```

本轮新增重点：**Orchestration Layer = Moxt 的真正操作系统内核**。

## 三、AI 同事角色设计：新增调度型 AI

### Orchestrator AI（调度控制者）

定位：负责“如何用 AI”，而不是“做什么”。

#### 核心职责

1. 决定是否采用单 agent、多 agent 并行或分阶段执行。
2. 决定 agent 组合、执行顺序、budget 分配。
3. 管理 Solution Set、early stop、fallback 策略。

#### 工具权限

- 读历史 performance（agent 成功率）。
- 控制 delegation。
- 控制 budget。
- 无代码写权限。

#### 协作边界

- 不参与具体实现。
- 不参与最终发布决策。

#### 评价指标

- 最优解命中率。
- 平均成本 / 任务。
- latency（任务完成时间）。
- 不必要并发率（浪费）。

关键升级：把“如何用 AI”从隐式经验，变成显式 AI 角色。

## 四、AI 原生工作空间设计：解空间与调度结构

### 新增核心结构：Solution Graph

不是 TaskGraph，而是：

```text
Task
 ├── Solution A (Agent 1)
 ├── Solution B (Agent 2)
 ├── Solution C (Agent 3)
```

每个 Solution 附带：cost、risk、verification score。

### 新增对象：Selection Policy

定义如何选：

```yaml
selection_policy:
  objective:
    - quality_weight
    - cost_weight
    - latency_weight
  constraints:
    - must_pass_tests
    - risk_below_threshold
```

### 新增：Adaptive Budget

不是固定预算，而是：

```text
初始预算 → 根据结果动态扩展或终止
```

## 五、新作业范式

从 AI 执行任务，进化为系统搜索解空间。

### 新范式

```text
Problem → Generate Solutions → Evaluate → Select → Refine
```

| 传统研发 | 新范式 |
|---|---|
| 写代码 | 搜索解 |
| review | 排序 |
| debug | 重新采样 |

本质变化：研发变成“搜索问题”而不是“构造问题”。

## 六、效率提升机制

### 1. 解空间并行

多个 agent 并行，不再受单路径限制。

### 2. Early Stop 机制

当发现一个解足够好，停止其他 agent，节省成本并降低延迟。

### 3. 动态 agent 选择

不同任务自动选择快模型、强模型、专用 agent。

### 4. 历史表现驱动优化

系统学习哪个 agent 在什么任务更强，并自动优化调度。

## 七、风险与边界

### 1. 组合爆炸（Combinatorial Explosion）

多 agent + 多解会导致成本指数级增长。必须限制并发并强制 early stop。

### 2. 错误最优解（Local Optimum）

系统可能选了“看起来最好”的解，但不是全局最优。需要保留探索概率并引入随机性。

### 3. 评价函数错误（Objective Misalignment）

如果 selection policy 错误，系统会稳定选错。

### 4. 调度过拟合

系统过度依赖历史，对新问题失效。

## 八、阶段性结论

最终模型升级为：

> Moxt = 一个用于“搜索、评估并选择软件实现方案”的 AI 编排系统。

其核心能力不在写代码，而在生成多个方案、评估这些方案、选择最优方案。

## 九、下一轮研究方向

1. 设计 Solution Scoring Function（评分函数）：如何量化“好代码”。
2. 构建 Agent Performance Model：不同 agent 的能力分布。
3. 推导 Exploration vs Exploitation 策略：何时尝试新 agent。
4. 设计 Verification Pipeline DSL：标准化验证流程。
5. 深入 Human-in-the-loop 最优插入点：人类在哪一步最有价值。
