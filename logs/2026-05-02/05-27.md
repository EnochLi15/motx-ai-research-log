# MOTX / Moxt 软件研发范式增量研究记录：Economic Layer 与资源优化

时间：2026-05-02 05:27

## 本轮新增发现

前几轮已经形成三层核心能力：

```text
Execution（执行）
Convergence（收敛）
Strategy（全局优化）
```

但本轮发现一个尚未建模的关键变量：

> **不同任务、不同 agent、不同验证路径，其“成本结构完全不同”，而当前系统默认“资源无限或等价”。**

现实中存在：

1. 不同 agent 成本差异巨大（token / 时间 / infra）
2. 高验证等级任务成本指数级上升
3. 多轮 Task Evolution 会显著放大成本
4. 多 agent 并行带来吞吐提升，但也带来资源浪费风险

因此，仅优化成功率 / 收敛效率 / 全局质量是不够的，还必须引入：

```text
Cost-aware Optimization（成本感知优化）
```

引出本轮核心新增概念：

> **Economic Layer（经济层）**：让整个研发系统在“质量-速度-成本”之间进行动态权衡。

## MOTX / Moxt 产品理解更新

在 9 层基础上，新增第 10 层：

```text
10. Economic & Resource Allocation Layer（经济与资源分配层）
```

### Economic Layer 核心作用

不是简单“算成本”，而是：

```text
为每一个 Task / Agent / Verification / Strategy 决策分配最优资源
```

### 三个核心对象（新增）

#### 1. Task Budget（任务预算）

```yaml
task_budget:
  task_id: TASK-123
  max_cost:
    tokens: 2M
    runtime_minutes: 60
    infra_cost: $3
  priority: high
  deadline: 2h
```

#### 2. Agent Cost Profile（Agent 成本画像）

```yaml
agent_profile:
  codex:
    cost_per_1k_tokens: 0.002
    avg_success_rate: 0.78
    avg_latency: low
  claude_code:
    cost_per_1k_tokens: 0.006
    avg_success_rate: 0.86
    avg_latency: medium
```

#### 3. Verification Cost Model

```yaml
verification_cost:
  unit_test: low
  integration_test: medium
  scenario_simulation: high
  full_e2e: very_high
```

## AI 同事角色设计更新

### 新增角色：Resource Allocator / 资源分配 AI

#### 定位

在任务执行前，决定“花多少资源做这件事”。

#### 核心职责

- 为 Task 分配 agent 类型、并行度、验证等级。
- 动态调整：如果 Task 收敛困难则增加预算；如果低价值任务则降低资源投入。
- 控制：防止 agent 无限重试，防止高成本验证滥用。

#### 决策逻辑

```text
Expected Value = 成功收益 × 成功概率
Expected Cost = agent cost + verification cost + retry cost
执行条件：Expected Value > Expected Cost
```

#### 工具权限

- 读 Task priority、历史成功率、agent performance。
- 写 task budget、agent assignment（建议）。
- 可限制最大 retry 次数、最大并行 agent 数。

#### 评价指标

- 单位成本成功率（success per cost）
- 资源浪费率
- 高价值任务完成率
- budget 超支率

### Strategy Planner 升级

新增维度：不仅优化质量，还要优化成本结构。例如：是否值得做某次重构？是否值得跑 full e2e？是否值得多 agent 竞争？

## AI 原生工作空间设计更新

### 新增核心对象：Execution Plan（执行计划）

在 Task 执行前生成：

```yaml
execution_plan:
  task_id: TASK-123
  agent_selection:
    primary: codex
    fallback: claude_code
  parallelism: 2
  verification_level: medium
  budget:
    max_tokens: 1.2M
    max_runtime: 40m
  strategy:
    retry_on_fail: true
    max_retries: 2
    escalate_to_human: if cost > threshold
```

### 新增机制：Dynamic Budget Adjustment

在执行过程中：

```text
if convergence_slow:
  increase budget
if repeated_failure:
  stop execution
if high_value_task:
  allow higher cost
```

## 软件研发新作业范式更新

从上一轮：

```text
Goal → Converge → Optimize
```

升级为：

```text
Goal
 → Spec
 → Execute (with budget)
 → Verify
 → Evolve
 → Optimize Strategy
 → Allocate Resources
 → 全局最优（质量 × 速度 × 成本）
```

### 新范式核心特征

1. 每个任务都有“成本意识”。
2. 不同任务采用不同执行策略。
3. 系统自动在“便宜但不稳” vs “昂贵但可靠”之间选择。

## 效率提升机制

新增核心公式：

```text
True Efficiency = 有效交付价值 / 总资源消耗
```

### Adaptive Fidelity（自适应精度）

不同任务采用不同精度：

| 任务类型 | 执行策略 |
|---|---|
| low risk bug | 单 agent + 低验证 |
| feature | 单 agent + 中验证 |
| refactor | 多 agent + 高验证 |
| migration | 人类 + 强验证 |

### Early Stop

如果系统判断：

```text
继续执行成本 > 预期收益
```

则终止任务或转人工。

### Speculative Execution（投机执行）

对高价值任务，多 agent 并行尝试不同方案并提前选优，前提是必须在预算内。

## 风险与边界

### 风险一：过度优化成本导致质量下降

Economic Layer 过强可能选择“便宜但错误”的路径。

对策：质量约束必须优先于成本约束。

### 风险二：预算分配不公平

高频小任务可能占用大量资源，高价值大任务资源不足。

对策：引入 priority weight 和长期价值评估。

### 风险三：成本模型不准确

agent cost 或 success rate 估算错误会导致决策失效。

对策：持续学习 agent performance（在线更新）。

## 阶段性结论

AI 软件研发系统的最终形态不是“自动化开发系统”，而是“资源优化的工程系统”。

三大核心维度必须同时成立：

```text
1. 正确性（Verification）
2. 收敛性（Task Evolution）
3. 经济性（Economic Layer）
```

进一步抽象：

```text
Moxt = 工程问题求解系统 + 资源优化系统
```

或者：

```text
Moxt 是一个“在有限资源下逼近最优解”的系统。
```
