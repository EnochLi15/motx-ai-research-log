# Evidence-Gated Agentic Development 与 AI 变更控制系统

时间：2026-05-02 16:27

## 本轮新增发现

本轮重点从“AI 编码能力”转向“AI 研发协作失败模式”。最新外部信号表明，AI coding agent 已经从辅助补全进入“读任务、改代码、跑测试、提 PR”的阶段，但真正瓶颈不在单个 agent 会不会写代码，而在 **任务边界、协作协议、证据验证、安全治理和人类审查负载**。

几个关键发现：

1. **AI 生成 PR 的失败并非主要因为不会写代码，而是因为协作语义不完整。**
   MSR 2026 相关研究分析了约 33k 个 agent-authored PR，发现文档、CI、构建更新类任务合并成功率较高，而性能优化和 bug fix 成功率较差；未合并 PR 通常改动更大、触碰文件更多、CI 不通过，并出现重复 PR、错误需求理解、无意义 reviewer 互动等问题。

2. **agent 产物必须从“代码 diff”升级为“带证据的交付包”。**
   另一项大规模研究分析了 24,014 个已合并 agentic PR 和 5,081 个人类 PR，关注 lines added/deleted、commit 数、触碰文件、PR 描述与代码 diff 的一致性。这个方向说明未来评估 AI PR 不能只看是否通过测试，而要看“描述—变更—验证证据”是否一致。

3. **安全风险成为 autonomous PR 的硬门槛。**
   DryRun Security 的 2026 报告被多处引用：其测试中 30 个 AI agent PR 里有 26 个引入至少一个高危漏洞。虽然该结果需要结合测试任务和报告方法审慎解读，但足以说明 Moxt 不能把 Security Reviewer 放在普通 code review 之后，而应把安全约束前置到任务生成、权限预算、diff 评估和发布决策全过程。

4. **企业级 agent 平台正在走向“治理运行时”。**
   Microsoft 近期发布 Agent Governance Toolkit，强调 autonomous agent 已经开始执行订票、交易、写代码、管理基础设施等动作，因此需要 runtime security；其提到 OWASP 2026 Agentic Applications Top 10 包括 goal hijacking、tool misuse、identity abuse、memory poisoning、cascading failures、rogue agents 等风险。

5. **AI-native engineering 的组织问题开始显性化。**
   Amazon 内部推动 AI-native practices，并跟踪 2,100 多个工程团队的 AI 使用情况，目标是提高发布速度；但也出现工具冗余、缺少清晰指导、文化阻力和“AI sprawl”等问题。这个信号强化了 Moxt 的定位：不是再增加一个 coding 工具，而是提供跨工具的组织级控制面。

## MOTX/Moxt 产品理解更新

上一轮已将 Moxt 收敛为 **AI Engineering Control Workspace / Delegation & Verification Workspace**。本轮进一步更新为：

> Moxt 的核心不是“让 AI 自动写更多代码”，而是让软件组织可以安全、可控、可验证地消费 AI 生成的研发变更。

因此，Moxt 应避免被定位成：

- 代码编辑器；
- 单 agent 执行器；
- prompt 管理工具；
- 简单任务看板；
- AI 版 Jira。

更准确的定位应是：

> **Moxt = AI-native Engineering Change Control System**
> 面向 AI 生成变更的任务定义、权限控制、执行编排、证据收集、审查分流、风险治理和组织记忆系统。

新的核心问题不再是：

> “哪个 agent 写代码最好？”

而是：

> “在什么任务、什么风险、什么上下文、什么验证证据下，哪个 agent 的哪个变更可以被接受？”

这意味着 Moxt 应该围绕 **Change Acceptance** 而不是 **Code Generation** 设计产品。

## AI 同事角色设计更新

本轮建议把 AI 同事进一步压缩为“变更控制责任链”，避免角色膨胀。

### 1. Intent Clarifier / 意图澄清 AI

**定位**：把用户、PM、设计、issue、Slack 讨论中的模糊意图转成可执行需求。

**职责**：

- 提炼目标、非目标、约束、验收标准；
- 识别需求缺口；
- 判断是否适合 AI 自动执行；
- 生成初版 `IntentSpec`。

**工具权限**：

- 可读：issue、文档、设计稿、历史 PR、Slack/会议摘要；
- 不可写代码；
- 不可修改任务状态为完成；
- 可创建 clarification comment 或需求补充建议。

**评价指标**：

- 后续返工率；
- 被实现 agent 误解的比例；
- 需求遗漏召回率；
- 人类补充问题数量。

### 2. Change Planner / 变更规划 AI

**定位**：把需求转成可委派的变更计划，而不是直接写代码。

**职责**：

- 生成 `ChangePlan`；
- 拆分任务边界；
- 标记风险区域；
- 选择执行模式：single-agent、multi-agent、human-led、review-only；
- 生成 `DeliveryContract` 和 `EvidenceRequirement`。

**工具权限**：

- 可读代码库结构、依赖图、历史缺陷；
- 可创建任务、分支计划、验证计划；
- 不可直接 merge；
- 高风险任务必须触发人类确认。

**评价指标**：

- 计划准确率；
- 任务拆分后冲突率；
- agent 执行失败率；
- plan 与最终 diff 的偏差。

### 3. Implementation Worker / 实现 AI

**定位**：可替换的 coding agent worker，可以是 Claude Code、Codex、Cursor、Copilot、Devin、内部 agent 等。

**职责**：

- 根据 `ChangePlan` 和 `DeliveryContract` 修改代码；
- 运行本地测试；
- 提交候选 diff；
- 生成实现说明。

**工具权限**：

- 只允许在授权 worktree / sandbox 中修改；
- 默认不能访问生产密钥；
- 默认不能直接推 main；
- 不能自评“完成”，只能提交 candidate。

**评价指标**：

- Contract 满足率；
- 测试通过率；
- diff 最小性；
- 回滚容易度；
- 缺陷注入率；
- reviewer 返工次数；
- 同类任务历史表现。

### 4. Evidence Builder / 证据构建 AI

**定位**：把 agent 的代码变更转成可审查、可机器消费的证据包。

**职责**：

- 收集测试结果；
- 对照验收标准；
- 生成 diff summary；
- 标注风险文件；
- 生成 `EvidencePack`；
- 检查 PR 描述与实际 diff 是否一致。

**工具权限**：

- 可读 diff、CI、测试报告、静态分析、安全扫描；
- 可触发测试和扫描；
- 不可修改业务代码；
- 不可批准发布。

**评价指标**：

- 证据完整率；
- 证据与 diff 一致性；
- 被 reviewer 追问次数；
- 漏报风险数；
- 误报率。

### 5. Risk & Security Sentinel / 风险安全 AI

**定位**：专门审查安全、权限、数据、迁移、支付、认证、发布风险。

**职责**：

- 识别 auth、permission、billing、migration、PII、secret、infra 等敏感变更；
- 生成 `RiskProfile`；
- 判断是否需要人工审批；
- 阻断不满足安全证据的变更。

**工具权限**：

- 可读代码、配置、IaC、依赖、权限策略；
- 可运行 SAST、dependency scan、secret scan；
- 可阻断 merge；
- 不可直接修改安全策略，除非在人类批准下。

**评价指标**：

- 高危漏洞漏报率；
- 风险误报率；
- 被人工 override 的比例；
- 发布后安全事件关联率。

### 6. Release Governor / 发布治理 AI

**定位**：决定一个 AI 生成变更是否可以进入 merge / release 阶段。

**职责**：

- 消费 `EvidencePack`；
- 对照 `DeliveryContract`；
- 检查风险预算；
- 判断是否 approve、request changes、escalate、reject；
- 维护 `ReleaseDecision`。

**工具权限**：

- 可读全部 evidence；
- 可设置 PR label / status check；
- 低风险变更可自动 approve；
- 中高风险必须升级给人类；
- 不应拥有直接部署生产权限。

**评价指标**：

- 错误放行率；
- 不必要阻断率；
- 审查耗时；
- 发布后回滚率；
- 人类 reviewer 节省时间。

## AI 原生工作空间设计更新

Moxt 工作空间应围绕“AI 变更生命周期”组织，而不是围绕聊天窗口组织。

### 新增核心对象

```text
IntentSpec
- user_goal
- non_goals
- constraints
- acceptance_criteria
- ambiguity_list
- source_refs

ChangePlan
- target_components
- expected_files
- forbidden_files
- dependency_map
- rollout_strategy
- rollback_strategy

DeliveryContract
- required_behavior
- required_tests
- required_non_regression
- security_requirements
- observability_requirements
- documentation_requirements

EvidencePack
- diff_summary
- test_results
- static_analysis
- security_scan
- contract_check
- risk_profile
- reviewer_notes
- known_limitations

ReleaseDecision
- decision: approve / request_changes / escalate / reject
- rationale
- violated_contract_items
- accepted_risks
- human_approver
- follow_up_tasks

AgentPerformanceRecord
- task_type
- repo_area
- risk_level
- agent_used
- outcome
- review_iterations
- defects_found
- post_merge_incidents
```

### 工作空间核心视图

1. **Change Control Board**
   展示所有 AI 生成变更的状态：planned、running、candidate、evidence_ready、under_review、approved、rejected、merged。

2. **Evidence-first PR View**
   PR 页面默认不先展示代码，而是展示：需求是什么、承诺交付什么、实际改了什么、哪些证据证明它完成、哪些风险仍然存在。

3. **Agent Reputation Dashboard**
   不展示泛泛的“哪个模型强”，而是展示：某 agent 在 bug fix 上表现如何、在 auth 相关代码上是否高风险、在前端样式任务上是否稳定、在大 diff 上是否容易失败。

4. **Risk Heatmap**
   按 repo 区域、文件类型、业务域展示 AI 变更风险。例如 `auth/` 高敏、`billing/` 高敏、`docs/` 低敏、`tests/` 中低敏、`migration/` 高敏。

## 软件研发新作业范式

本轮建议将 Moxt 的研发范式定义为：

> **Evidence-Gated Agentic Development**

即：

```text
需求不是直接派给 AI 写代码，
而是先变成可验证的交付契约；
AI 不是提交“完成结果”，
而是提交候选变更；
系统不是相信 agent 自评，
而是要求 Evidence Pack；
人类不是逐行审查所有代码，
而是审查风险、证据和异常。
```

### 新流程

```text
1. Intent Capture
   用户 / PM / issue / 文档输入目标

2. IntentSpec
   意图澄清 AI 结构化需求

3. ChangePlan
   变更规划 AI 拆解范围、风险和验证路径

4. Delegation
   Orchestrator 选择一个或多个 coding agents

5. Candidate Change
   agent 在隔离环境中生成候选变更

6. Evidence Build
   Evidence Builder 收集测试、扫描、diff、contract check

7. Risk Review
   Security Sentinel 做风险分级

8. Release Decision
   Release Governor 决定 approve / reject / escalate

9. Outcome Memory
   结果进入 AgentPerformanceRecord 和组织记忆
```

### 人类角色变化

人类工程师不再主要做：手动写所有 boilerplate、手动跑重复测试、手动整理 PR 描述、手动同步需求上下文。

人类工程师转向：设定边界、审查高风险决策、处理复杂架构权衡、定义团队规范、校正 agent 行为、维护长期技术方向。

## 效率提升机制

Moxt 的效率提升不应只来自“代码写得更快”，而应来自六类机制：

### 1. 降低任务启动成本

通过 IntentSpec、ChangePlan、上下文包自动生成，让 agent 不再每次从零理解项目。

指标：issue → first candidate PR 时间、人类补充上下文次数、agent 首轮失败率。

### 2. 降低审查成本

通过 EvidencePack 让 reviewer 从“读全部 diff”转为“看风险摘要 + 验证证据 + 异常项”。

指标：PR review 平均耗时、reviewer comment 数、人类只需深看代码的比例。

### 3. 降低返工成本

DeliveryContract 提前定义验收标准，减少 agent 做错方向。

指标：request changes 次数、合并前迭代轮次、需求误解导致的废弃 diff 数。

### 4. 降低安全事故成本

Security Sentinel 前置介入，防止 AI 快速制造隐性安全债。

指标：高危变更自动拦截数、secret / permission / auth 漏报数、发布后安全缺陷数。

### 5. 提高 agent 选择效率

AgentPerformanceRecord 让系统根据任务类型选择 agent，而不是固定使用一个模型。

指标：agent-task 匹配成功率、同类任务平均成本、同类任务平均通过率。

### 6. 形成组织学习

每个任务结果都沉淀到 Outcome Memory，下一次任务可复用策略。

指标：重复问题减少率、相似任务耗时下降、agent prompt / policy 自动优化效果。

## 风险与边界

本轮最重要的风险判断：

> AI 研发系统的主要风险不是“AI 不够聪明”，而是“AI 变更进入工程系统的速度超过了组织验证能力”。

### 需要重点防范

1. **AI sprawl**：工具太多、规范太少，团队不知道什么时候用哪个 agent。
2. **Reviewer overload**：agent 生成 PR 太快，人类 reviewer 被淹没，最终审查质量下降。
3. **False confidence**：PR 描述很完整、测试也通过，但实际没有满足业务目标。
4. **Security blindness**：agent 默认优化“功能可用”，不一定理解权限、数据泄露、注入、越权、审计等安全约束。
5. **Large diff failure**：未合并 agent PR 往往触碰更多文件、改动更大；Moxt 应对大 diff 默认拆分或升级。
6. **Memory poisoning**：如果错误结论进入组织记忆，会在后续任务中被反复放大。
7. **Autonomous merge risk**：低风险场景可以自动合并，但 auth、billing、infra、migration、data deletion 等场景必须人工审批。

## 阶段性结论

本轮结论：

> Moxt 的最强产品机会，是成为 AI 生成软件变更的“控制塔”和“证据门禁”，而不是成为另一个代码生成入口。

更具体地说：

```text
Moxt = IntentSpec
     + ChangePlan
     + Delegation Control
     + Agent Sandbox
     + EvidencePack
     + RiskProfile
     + ReleaseDecision
     + OutcomeMemory
```

AI 同事设计应从“模拟人类研发岗位”进一步演进为“围绕变更接受的责任链”：

```text
Intent Clarifier
→ Change Planner
→ Implementation Worker
→ Evidence Builder
→ Risk & Security Sentinel
→ Release Governor
```

这一版比早期 18 个 AI 同事更适合作为产品 MVP，因为它直接围绕企业最愿意付费的问题：

- AI 写代码太快，如何管住？
- AI PR 太多，如何审？
- 哪些可以自动合并，哪些必须人工看？
- 哪个 agent 在什么任务上可信？
- 如何证明 AI 真的完成了需求？
- 如何降低 AI 引入安全和质量事故的概率？

## 下一轮研究方向

1. **EvidencePack schema 设计**：明确定义测试结果、diff 摘要、安全扫描、contract check、风险接受项的机器可读结构。
2. **ReleaseDecision 策略矩阵**：按风险等级、代码区域、测试信心、agent reputation 决定自动放行、请求修改、人工审批或拒绝。
3. **AgentPerformanceRecord 评分模型**：设计不同任务类型下的 agent reputation：bug fix、docs、frontend、backend、infra、security、migration。
4. **Large Diff Control Protocol**：针对大规模 agent PR 设计自动拆分、分阶段验证、局部回滚和 reviewer 分配机制。
5. **Moxt 与 GitHub Checks 的落地映射**：将 EvidencePack、RiskProfile、ReleaseDecision 映射为 GitHub check run、PR label、required status、review comment。
6. **Memory Safety 机制**：研究哪些内容可以进入长期记忆，哪些必须标记为 tentative / expired / human-verified。
