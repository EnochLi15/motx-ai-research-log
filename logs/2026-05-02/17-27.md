# Agent Identity Boundary 与非人类身份治理

时间：2026-05-02 17:27

## 本轮新增发现

本轮重点从“AI 变更是否有证据”继续推进到一个更底层的问题：**AI coding agent 不是普通工具调用者，而是新的非人类工程身份。**

GitHub Copilot coding agent 已经支持从 GitHub Issues、VS Code、JetBrains IDE、GitHub CLI、agents panel 等入口创建 PR，并在后台工作、提交 draft PR、等待人类 review。GitHub 文档也明确区分了 coding agent 与 IDE 内 agent mode：coding agent 更接近异步工程贡献者，而不是交互式补全工具。

GitHub 在发布 Copilot coding agent 时强调，它运行在 GitHub 原生控制层之内，使用 draft PR、session logs、branch protection、受控联网、人类批准后才运行 CI/CD workflow 等机制。这说明主流平台已经开始把 coding agent 纳入 SDLC 控制面，而不是仅仅把它当作编辑器插件。

新的企业风险信号也更清晰：企业采用 AI agents 的速度超过了治理和安全能力，传统 IAM 主要面向人类用户和静态权限，难以处理 agent 的连续运行、链式委派、上下文泄露、不可追踪代理链和非人类身份扩张问题。

研究侧也出现一个值得 Moxt 吸收的新方向：agent-authored PR 的 library usage 研究分析了 26,760 个 agent-authored PR，发现 agent 经常导入库，但很少新增依赖；新增依赖时多数会指定版本。这说明 agent 行为并非全然混乱，但仍需要被记录成结构化行为画像，而不是只看最终 PR 是否合并。

另一个 task-stratified PR acceptance 研究比较了 OpenAI Codex、GitHub Copilot、Devin、Cursor、Claude Code 等 agent 在不同任务类型上的 PR 接受情况。这强化了上一轮结论：Moxt 不应问“哪个 agent 最强”，而应问“哪个 agent 在哪个任务类型、哪个 repo 区域、哪个风险等级下可信”。

## MOTX/Moxt 产品理解更新

上一轮将 Moxt 定位为：

> AI-native Engineering Change Control System

本轮建议进一步补充：

> Moxt = Agentic Engineering Identity & Change Control Plane

也就是说，Moxt 不仅管理 AI 生成的变更，还要管理“谁”生成了变更、“以什么身份”生成、“继承了谁的权限”、“调用了哪些工具”、“是否发生过代理链委派”、“最终责任如何落到人类或团队”。

软件研发里的 AI agent 会越来越像一种新型工程身份：

```text
human engineer
service account
CI bot
deployment bot
AI coding agent
AI reviewer agent
AI release agent
```

过去的权限系统主要能回答：

```text
哪个人可以访问哪个 repo？
哪个 bot 可以触发哪个 workflow？
```

但 AI-native 研发系统必须回答：

```text
哪个 agent 可以在什么任务上下文里，
以什么权限预算，
访问哪些上下文，
修改哪些语义区域，
调用哪些工具，
生成什么级别的变更，
由谁承担最终责任？
```

因此，Moxt 的差异化不只是 EvidencePack，而是：

```text
Agent Identity
+ Delegation Chain
+ Semantic Permission
+ Evidence-Gated Change Acceptance
+ Agent Reputation
+ Human Accountability
```

## AI 同事角色设计更新

本轮建议在现有 6 个变更控制 AI 同事之外，新增一个横向治理角色：**Agent Identity Steward / AI 身份管理员**。

### Agent Identity Steward / AI 身份管理员

**定位**：管理所有 AI 同事和外部 coding agent 的身份、权限、委派链和责任边界。

**职责**：

- 为每个 agent 建立稳定身份；
- 记录 agent 类型：planner、worker、reviewer、security、release、external worker；
- 管理 agent 的默认权限模板；
- 记录 agent 是否可长期运行、是否可调用外部工具、是否可访问私有上下文；
- 维护 delegation chain；
- 检查 agent 是否越权；
- 将每次变更绑定到 agent identity 与 human accountable owner。

**工具权限**：

- 可读 agent registry、repo policy、权限策略、审计日志；
- 可暂停 agent session；
- 可降级 agent 权限；
- 可要求重新授权；
- 不应直接修改业务代码；
- 不应直接 approve release。

**协作边界**：

- 不替代 Security Sentinel；
- 不判断代码是否正确；
- 只判断“这个 agent 是否有权做这件事”；
- 与 Release Governor 协作，作为发布决策的身份治理输入。

**评价指标**：

- agent 越权拦截率；
- 未归属 agent action 数量；
- delegation chain 可追踪率；
- 高风险权限误授率；
- agent session ghost rate；
- human accountable owner 覆盖率。

### 对现有角色的补充调整

现有责任链可以改为：

```text
Intent Clarifier
→ Change Planner
→ Agent Identity Steward
→ Implementation Worker
→ Evidence Builder
→ Risk & Security Sentinel
→ Release Governor
```

其中 Agent Identity Steward 在 Delegation 之前介入，回答：

```text
这个任务是否可以交给 agent？
交给哪个 agent？
这个 agent 需要什么临时权限？
权限多久过期？
是否需要 human owner？
是否允许调用 MCP / CI / package registry / cloud resource？
```

## 软件研发新作业范式更新

上一轮范式是：

> Evidence-Gated Agentic Development

本轮建议扩展为：

> Identity-Bound Evidence-Gated Agentic Development

也就是：

```text
AI 不能只是提交候选变更；
AI 必须以明确身份提交候选变更。

系统不能只验证代码和测试；
系统还必须验证该 agent 是否有权执行该变更。

发布不能只看 EvidencePack；
还要看 DelegationChain、PermissionGrant 和 accountable owner。
```

新的流程：

```text
1. Intent Capture
2. IntentSpec
3. ChangePlan
4. Agent Identity Check
5. PermissionGrant
6. Delegation to Worker Agent
7. Candidate Change
8. EvidencePack
9. RiskProfile
10. ReleaseDecision
11. OutcomeMemory
12. AgentPerformanceRecord 更新
```

关键变化是：**Delegation 不再只是调度问题，而是身份治理问题。**

## 效率提升机制

本轮识别出的效率提升不是“让 agent 更快写代码”，而是减少组织采用 agent 后的治理摩擦。

### 1. 降低授权沟通成本

团队不需要每次问“这个 AI 能不能改这个模块”，而是通过 AgentIdentity + PermissionGrant 自动判定。

### 2. 降低安全 review 成本

Security Sentinel 不需要从零判断上下文，可以直接消费：

```text
agent identity
permission grant
semantic scope
delegation chain
evidence pack
```

### 3. 降低 agent sprawl 成本

当团队同时使用 Copilot、Codex、Claude Code、Cursor、Devin、内部 agent 时，Moxt 可以统一记录它们在不同任务类型上的表现。

### 4. 提高责任追踪效率

每个 agent action 都绑定 accountable human owner，避免出现“AI 做的，所以没人负责”的组织漏洞。

### 5. 提高自动化放行比例

低风险任务如果满足以下条件，可以更安全地自动放行：

```text
可信 agent
低风险 repo 区域
权限未越界
EvidencePack 完整
RiskProfile 低风险
历史同类任务表现稳定
```

## 风险与边界

本轮新增的核心风险是：

> AI agent 的失败不只是代码失败，也可能是身份、权限和委派链失败。

重点风险包括：

1. **Agent identity abuse**：agent 获得超过任务所需的权限。
2. **Delegation opacity**：人类只看到最终 PR，看不到中间 agent chain。
3. **Permission inheritance leak**：agent 继承了人类或工具的过大权限。
4. **Context overexposure**：低风险任务却读取了高敏文档、secret、客户数据或内部策略。
5. **Tool misuse**：agent 在 coding task 中调用了 package publish、cloud infra、production DB 等高风险工具。
6. **Accountability gap**：PR 由 AI 创建，但组织内没有明确 owner。
7. **Reputation laundering**：一个低信誉 agent 通过高信誉 planner 或 wrapper 进入执行链。
8. **Long-running ghost agent**：任务结束后 agent session、token、权限或 sandbox 未被回收。

## 阶段性结论

Moxt 的产品机会进一步清晰：

```text
Moxt 不只是 AI 变更控制系统，
而是 AI 工程身份 + 权限 + 证据 + 发布决策控制面。
```

当前最强定位可以表述为：

> Moxt is the control plane for accepting AI-generated software changes safely.

更完整的内部定义：

```text
Moxt = AgentIdentity
     + PermissionGrant
     + DelegationChain
     + IntentSpec
     + ChangePlan
     + DeliveryContract
     + Agent Sandbox
     + EvidencePack
     + RiskProfile
     + ReleaseDecision
     + OutcomeMemory
     + AgentPerformanceRecord
```

本轮最重要判断：

> 企业真正缺的不是更多 coding agents，而是一个能回答“哪个 AI 身份在什么授权边界内完成了什么变更，并凭什么可以被接受”的系统。

## 下一轮研究方向

1. 设计 `AgentIdentity` schema 与 agent registry。
2. 设计 `PermissionGrant` 的语义权限模型，区分 file scope、tool scope、data scope、runtime scope。
3. 设计 `DelegationChain` 可视化：从 human intent 到 release decision 的完整责任链。
4. 设计 agent token / session TTL / revocation 策略。
5. 设计 Agent Reputation 与 AgentIdentity 的关系：身份是“谁”，声誉是“过去表现如何”。
6. 设计 Moxt 与 GitHub branch protection、required checks、Copilot session logs、MCP tool grants 的集成方式。
7. 研究非人类身份治理在软件研发组织里的最佳实践，并映射到 Moxt 的企业版能力。
