# Verification Pipeline DSL 与验证编排层（Verification Orchestration Layer）

时间：2026-05-02 19:27

## 本轮新增发现

在上一轮已经建立：

```text
Identity → Permission → Sandbox → ExecutionTrace → Evidence → Decision
```

但仍然存在一个关键断点：

> **EvidencePack 的生成过程仍然是隐式、分散、不可编排的。**

当前主流问题：

```text
- test 是 CI 跑的
- lint 是另一个 job
- security scan 是第三方工具
- contract check 可能不存在
- review comment 是人类/AI离散行为
```

这些“验证行为”没有统一模型，导致无法标准化 EvidencePack，无法自动化 ReleaseDecision，也无法根据任务类型动态调整验证强度。

因此本轮核心新增：

> **Verification Pipeline DSL（验证流水线语言） + Verification Orchestration Layer**

## MOTX/Moxt 产品理解更新

产品模型升级为四层：

```text
Layer 1: Identity & Permission
Layer 2: Execution Isolation (Sandbox)
Layer 3: Verification Orchestration
Layer 4: Acceptance & Governance
```

核心补充：

> Moxt 不只是收集证据，而是**定义证据是如何被生产出来的**。

新的产品本质：

> **Moxt = 定义“AI 变更必须经过什么验证流程才可被接受”的系统**

## AI 同事角色设计更新

新增一个核心 AI 角色：

### Verification Architect / 验证架构师

**定位**：根据任务类型、风险等级、代码区域，动态生成验证流水线（Verification Pipeline）。

### 职责

- 为每个任务生成 VerificationPipeline；
- 定义验证步骤（steps）：test、lint、type check、security scan、contract check、semantic diff analysis；
- 配置验证强度（light / standard / strict）；
- 定义失败策略：block、retry、degrade、escalate；
- 定义 EvidencePack schema；
- 为 ReleaseDecision 提供结构化输入。

### 工具权限

- 可读取 TaskGraph、RiskProfile、repo metadata；
- 可调用 CI system、static analysis tools、test runners；
- 不可修改业务代码；
- 不可 approve PR。

### 协作边界

- 不执行具体测试（交给执行层）；
- 不做最终决策（交给 Release Governor）；
- 不管理权限（交给 Identity Steward）。

### 评价指标

```text
verification coverage rate
false negative rate（漏检）
false positive rate（误杀）
pipeline execution cost
verification latency
release decision accuracy contribution
```

## 软件研发新作业范式更新

范式再次升级：

> **Programmable Verification-Gated Agentic Development**

关键变化：

### 1. 验证流程成为“可编排对象”

以前：

```text
CI = 固定 pipeline
```

现在：

```text
每个 Task → 自定义 VerificationPipeline
```

### 2. 不同任务类型对应不同验证策略

```text
bug fix → 轻量 + regression test
refactor → semantic diff + test
security fix → strict + scan + audit
migration → staged validation
```

### 3. EvidencePack = Pipeline 输出

```text
EvidencePack = VerificationResult + ExecutionTrace + RiskProfile
```

### 4. ReleaseDecision 可自动化

因为输入已经结构化，包括 VerificationResult、RiskProfile、AgentReputation。

## 效率提升机制

### 1. 避免过度验证（Over-Verification）

不同任务用不同 pipeline，小改动不等于全量 CI，从而节省大量时间和成本。

### 2. 提高自动放行比例

如果 pipeline 全通过、风险低、agent 可信，则可进入自动 merge 候选。

### 3. 减少人工 review 负担

reviewer 不再从零判断质量，而是判断验证 pipeline 输出是否合理。

### 4. 支持验证演化（Learning Loop）

系统可以学习哪些验证步骤真正有效，哪些可以裁剪。

## 风险与边界

### 1. Pipeline 设计错误

验证覆盖不足会导致错误被放行。

### 2. 验证成本膨胀

复杂 pipeline 可能运行时间过长、资源消耗过高。

### 3. Tool 依赖风险

外部 scan tool 可能不稳定或版本不一致。

### 4. “验证即真理”的误区

即使 pipeline 通过，仍可能存在逻辑问题、产品问题和隐性风险。

## 阶段性结论

Moxt 的系统已经演化为：

```text
Intent
→ Plan
→ Identity
→ Permission
→ Sandbox
→ Execution
→ Verification Pipeline
→ Evidence
→ Decision
```

关键认知升级：

> 软件研发的核心不是“写代码”，而是“定义在什么验证规则下代码可以被接受”。

最终产品本质：

> **Moxt = Software Change Acceptance Programming System**
