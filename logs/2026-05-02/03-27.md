# MOTX / Moxt 软件研发范式增量研究记录：Adaptive Task-Verification Loop

时间：2026-05-02 03:27

## 本轮新增发现

上一轮已经提出 **Verifiable Delivery Protocol**（可验证交付协议），解决“PR 是否真的完成目标”的问题。本轮进一步发现一个更深层结构性问题：

> 当前系统默认“Task 是正确的，只需要验证 Delivery”；但现实中，大量失败来自 **Task 本身设计错误或不充分**。

具体表现：

1. Requirement 不完整 → agent 正确实现错误需求
2. TaskGraph 拆分错误 → 局部正确但整体失败
3. 验证标准错误 → 系统判定“成功”，用户仍然不满意
4. 多轮返工 → 不是实现问题，而是任务定义逐步收敛

因此，仅有：

```text
Task → Execution → Verification
```

是不够的，还需要：

```text
Task → Execution → Verification → Task 修正 → 再执行
```

这引出本轮核心新增概念：

> **Adaptive Task-Verification Loop（自适应任务-验证闭环）**

## MOTX / Moxt 产品理解更新

在已有 7 层架构上，本轮新增一个关键系统能力：

```text
8. Task Feedback & Evolution Layer（任务反馈与进化层）
```

### 核心升级

Moxt 不只是 structuring tasks、orchestrating agents、verifying delivery，而是持续修正 Task 本身，使系统收敛到正确问题定义。

### 三类失败分类

Moxt 必须区分三种失败来源：

```text
1. Execution Failure（执行失败）
   → agent 没做好
2. Verification Failure（验证失败）
   → 验证规则或覆盖不足
3. Task Failure（任务失败）
   → 任务定义错误或不完整
```

当前行业几乎只优化第 1 类。Moxt 的机会在于系统性识别第 3 类失败，并自动修正 Task。

## AI 同事角色设计更新

### 新增角色：Task Critic / 任务批判 AI

定位：专门识别“任务是否定义正确”的 AI，同层级于 Task Architect，但职责相反。

核心职责：

- 分析失败 PR / 被拒 PR / 反复修改任务
- 判断失败是否来自 Task scope 错误、Requirement 不完整、验收标准错误
- 识别同一任务多次修改、不同 agent 反复失败、Reviewer/QA/Auditor 结论冲突
- 输出 Task Diagnosis Report：failure_type、root_cause、suggested_task_fix

工具权限：

- 读 TaskGraph、PR 历史、review comments、auditor verdict、QA 结果
- 写 task patch proposal（建议修改 Task，而不是代码）
- 禁止修改代码、直接执行任务

评价指标：

- Task 修正后成功率提升
- 减少重复失败次数
- Task 重写质量（被采纳率）
- 误判率（把 execution 当 task）

### Task Architect 升级

新增职责：不仅生成 Task，还要接收 Task Critic 的反馈并重构 TaskGraph。

形成：

```text
Task Architect ←→ Task Critic
```

一个生成，一个批判，形成闭环。

### Delivery Auditor 升级

新增输出：

```yaml
delivery_audit:
  result: fail
  failure_type: task | execution | verification
  confidence: 0.82
```

不再只说“失败”，而是说“为什么失败”。

## AI 原生工作空间设计更新

### 新增核心对象：Task Evolution Log

用于记录任务如何被不断修正。

```yaml
task_evolution_log:
  task_id: TASK-123
  versions:
    - v1:
        definition: ...
        result: fail
        failure_type: task
    - v2:
        changes:
          - refined acceptance criteria
          - reduced scope
        result: partial
    - v3:
        result: success
  convergence_steps: 3
  final_confidence: 0.91
```

### 新增对象关系

```text
Requirement
   ↓
Task (v1)
   ↓
Execution
   ↓
Verification
   ↓
Task Critic
   ↓
Task (v2)
   ↓
...
```

Moxt 从“静态任务系统”升级为动态任务进化系统。

## 软件研发新作业范式更新

上一轮：

```text
Goal → Spec → Execute → Verify
```

本轮升级为：

```text
Goal
 → Spec
 → Execute
 → Verify
 → Diagnose Failure
 → Evolve Task
 → Re-execute
 → Converge
```

关键变化：研发不再是一次性 pipeline，而是收敛过程（convergence process）。

### 新范式核心特征

1. Task 不再是静态输入，而是动态变量
2. 系统主动发现“问题定义错误”
3. 成功是“收敛”，而不是“通过测试”

## 效率提升机制

新增关键指标：

```text
Task Convergence Efficiency（任务收敛效率）
= 成功任务数 / 平均 Task 迭代次数
```

优化目标：减少无效 Task 版本、agent 重复尝试，增加每次 Task 修正的信息密度。

### Failure Compression

不是减少失败，而是让每次失败都最大化信息量：

- Delivery Auditor 输出结构化 failure 类型
- Task Critic 提取 root cause
- 自动生成 Task patch
- 下一轮执行直接基于修正

### Cross-Task Learning

不同任务的失败可共享：如果多个任务在 billing module 失败，自动提升该模块风险等级，Task Architect 自动收紧 scope，Security Sentinel 提高权限要求。

## 风险与边界

### 风险一：系统过度修改 Task（目标漂移）

Task Critic 过强可能不断“调整目标”，最终偏离原始 Goal。

对策：Goal 不可变；Task 修改必须引用 Goal；人类可锁定关键 Task。

### 风险二：收敛到局部最优解

系统可能快速找到“看起来可行”的方案，但不是最优或正确方案。

对策：引入 exploration budget，允许多种实现路径；引入对比执行（multi-agent solution）。

### 风险三：反馈回路过长

如果 Execution → Verification → Critic → Task → Execution 太慢，会拖慢整体效率。

对策：小任务快速收敛，高风险任务才走完整闭环，低风险任务跳过 Task Critic。

## 阶段性结论

软件研发的本质不是“把任务做对”，而是“把问题定义对，并逐步收敛”。

Moxt 的核心能力从 Task Orchestration + Delivery Verification 升级为：

```text
Task Evolution System（任务进化系统）
```

进一步抽象：

```text
研发系统 = 搜索问题空间 + 收敛到正确解
```

Moxt 本质上是在做：

```text
工程问题求解的操作系统（Problem-Solving OS）
```

## 下一轮研究方向

1. Task Failure 自动分类模型：如何区分 task vs execution vs verification？
2. Task Patch 自动生成：如何从 failure 自动生成更优 Task？
3. Convergence Strategy：如何控制探索 vs 收敛的平衡？
4. Multi-Agent Solution Competition：同一任务多 agent 并行，如何选最优解？
5. Task Complexity 评估模型：如何预测任务需要多少轮迭代？
