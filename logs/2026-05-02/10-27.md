# MOTX / Moxt 软件研发范式研究增量记录：Delegation Control Plane 与可验证委派层

时间：2026-05-02 10:27

## 本轮新增发现

本轮重点不是继续扩展 AI 同事数量，而是研究 **Moxt 如何在已有 TaskGraph Runtime、ActionIntent、RiskBudget、DeliveryContract、Agent Adapter 之上，形成一个可控的“工程代理委派系统”**。

外部趋势进一步确认：主流 coding agent 正在从 IDE 内补全，进入后台执行、Issue 委派、PR 生成、PR 修改、CI 修复、多任务并行阶段。OpenAI Codex 定位为云端软件工程 agent，可在独立云沙箱中并行处理多个任务，包括写功能、修 bug、回答代码库问题和提出 PR；GitHub Copilot coding agent 支持把 issue 分配给 Copilot 或要求它创建 / 修改 PR；Claude Code GitHub Actions 支持在 issue / PR 中通过 `@claude` 触发分析代码、创建 PR、实现功能和修复 bug。

这意味着 Moxt 的机会不应是再造一个 agent，而是成为 **多种外部 / 内部 coding agent 的工程控制面、委派层和验证层**。

## MOTX/Moxt 产品理解更新

上一轮结论是：Moxt 应把软件研发转化为可控的搜索与验证系统。本轮进一步推进为：

> Moxt 应成为 AI 工程组织里的 **Delegation Control Plane**：把任务拆成可委派单元，选择合适 agent 执行，把执行结果转为候选 PR / patch / decision，再通过验证、审计、风险控制和人类审批完成收敛。

外部 agent 越强，Moxt 越不应与它们直接拼模型能力，而应控制以下 6 件事：

1. 任务是否值得委派
2. 委派给哪个 agent / 哪类 agent
3. 给 agent 多少上下文、权限、预算和时间
4. agent 产物如何被验证
5. 多个候选方案如何比较
6. 最终决策如何进入组织记忆

因此，Moxt 的核心产品形态可以从 “AI-native engineering workspace” 进一步表述为：

> 面向研发组织的 AI Engineering Delegation & Verification Workspace。

## AI 同事角色设计增量

本轮不建议新增太多角色，而是把已有角色重新分层。

### A. 必须稳定存在的 5 个核心角色

#### 1. Task Architect / 任务架构师

定位：把模糊目标转为可执行 TaskGraph。

新增职责：

- 判断任务是否适合 agent 委派。
- 生成 `DelegationSpec`。
- 标记任务是否允许多 agent 并行尝试。
- 为每个节点定义上下文包、验收条件、风险预算和回滚要求。

核心指标：

- 任务拆解后返工率。
- 任务节点依赖错误率。
- 需求遗漏率。
- agent 执行前后任务定义变更次数。

#### 2. Agent Coordinator / 代理协调员

此前 Coordinator 更像调度员，本轮应升级为代理市场的 broker。

新增职责：

- 根据任务类型、代码区、历史表现、成本、风险选择 agent。
- 支持同一任务多候选 agent 并行。
- 维护 agent reputation。
- 决定是否重试、换 agent、拆小任务或升级给人类。

工具权限：

- 可启动外部 coding agent session。
- 可读取 agent 历史表现。
- 可分配预算。
- 不可直接 merge / release。

核心指标：

- 委派成功率。
- 平均 agent 尝试次数。
- 每类任务最优 agent 命中率。
- 单任务 token / compute 成本。
- 无效 PR 率。

#### 3. Implementation Engineer / 实现工程 AI

定位从“亲自编码”变成两种模式：

- direct executor：自己写代码。
- delegated executor：调用 Codex / Copilot / Claude Code / 内部 agent 执行。

新增职责：

- 生成候选实现。
- 修改失败候选。
- 根据 Reviewer / QA 反馈修复。
- 输出实现解释和风险说明。

权限边界：

- 可以写 branch / draft PR。
- 默认不能改 auth、billing、permission、migration、infra 等高风险区。
- 高风险区必须通过 Semantic Permission Layer 和 ActionIntent 审批。

#### 4. Delivery Auditor / 交付审计 AI

本轮地位上升。外部 agent 越多，Delivery Auditor 越关键。

新增职责：

- 对不同 agent 产物统一验收。
- 检查 PR 是否满足 DeliveryContract。
- 对候选 PR 打分。
- 识别“看起来完成但实际偏题”的实现。

评价指标：

- 虚假完成识别率。
- 漏检缺陷率。
- 与人类 reviewer 判断一致率。
- 回归问题提前发现率。

#### 5. Knowledge Steward / 知识管家 AI

新增职责：

- 把每次委派结果写入 agent reputation。
- 维护任务模式库。
- 记录哪些任务适合哪个 agent。
- 提炼失败案例和反模式。

评价指标：

- 后续任务复用率。
- 重复踩坑下降率。
- agent 选择准确率提升。
- 文档 / 规则过期率。

## 新增关键对象：DelegationSpec

建议在 Moxt 对象协议中加入：

```yaml
DelegationSpec:
  id: delegation_xxx
  task_id: task_xxx
  task_type: bugfix | feature | refactor | test | migration | doc | investigation
  delegation_mode: single_agent | multi_agent_competition | human_assisted | no_delegation
  candidate_agents:
    - codex
    - github_copilot
    - claude_code
    - internal_impl_agent
  selected_agent: codex
  context_package:
    repo_scope:
      - packages/auth/**
      - apps/web/**
    required_docs:
      - architecture.md
      - api-contract.md
    excluded_paths:
      - billing/**
      - infra/prod/**
  permission_budget:
    max_files_changed: 12
    max_diff_lines: 800
    allow_tests: true
    allow_dependency_change: false
    allow_schema_change: false
  risk_budget:
    semantic_risk: medium
    requires_human_approval: true
    rollback_required: true
  verification_contract:
    tests_required:
      - unit
      - integration
    checks_required:
      - lint
      - typecheck
      - security_scan
    delivery_contract_id: dc_xxx
  timeout:
    soft_minutes: 20
    hard_minutes: 45
  output_expected:
    - patch
    - draft_pr
    - implementation_notes
    - risk_notes
```

这个对象的价值是让 Moxt 不只是“叫 agent 干活”，而是把委派变成可控、可审计、可比较、可复盘的工程行为。

## AI 原生工作空间设计更新

建议新增一个核心视图：

# Agent Workbench / 代理工作台

它不是聊天窗口，而是任务委派与验证控制台。

每个任务节点应展示：

```text
Task Node
├── 当前状态
├── 绑定 agent
├── 委派上下文
├── 权限预算
├── 风险预算
├── 执行日志
├── 候选 PR / patch
├── 测试结果
├── DeliveryContract 满足度
├── Reviewer 反馈
├── Auditor 评分
└── 决策记录
```

对于多 agent 并行任务，展示方式应类似：

```text
Candidate A: Codex
- 成本：低
- 测试：通过
- diff：中等
- 风险：中
- Delivery score：0.82

Candidate B: Claude Code
- 成本：中
- 测试：通过
- diff：较小
- 风险：低
- Delivery score：0.88

Candidate C: Internal Agent
- 成本：低
- 测试：失败
- diff：大
- 风险：高
- Delivery score：0.43
```

Moxt 的决策不应是“谁先完成用谁”，而是：

> 谁在 DeliveryContract、风险、成本、可维护性、测试信心上的综合分最高，谁进入下一阶段。

## 软件研发新作业范式更新

建议把 Moxt 的研发范式定义为：

# Human sets intent, Moxt structures work, agents compete / execute, verification selects, humans approve exceptions.

中文表达：

> 人类定义目标，Moxt 结构化任务，AI 代理执行或竞争，验证层选择结果，人类处理高风险决策和例外。

对应流程：

```text
Goal
→ TaskGraph
→ DelegationSpec
→ Agent Session
→ Candidate Patch / PR
→ Verification Run
→ Delivery Audit
→ Human Review if needed
→ Merge / Release
→ Memory Update
→ Agent Reputation Update
```

这与传统研发流程的区别：

| 传统研发 | Moxt 范式 |
|---|---|
| 人分任务给人 | 系统把任务委派给最合适 agent |
| 单人实现 | 多 agent 可并行生成候选方案 |
| Code review 主要看代码 | Delivery audit 先看是否完成目标 |
| 测试是后置动作 | 验证契约在任务创建时生成 |
| 经验留在人脑 | 结果进入 agent reputation 和组织记忆 |
| 项目管理是状态同步 | 项目管理变成任务图运行时控制 |

## 效率提升机制

本轮认为，Moxt 的效率提升不只来自“AI 写代码更快”，而来自 5 个系统性杠杆：

### 1. 委派匹配效率

不同 agent 适合不同任务。Moxt 应长期学习：哪个 agent 擅长 bugfix、测试补全、前端 UI；哪个 agent 在大型 refactor 上失败率高；哪个 agent 容易过度修改。

### 2. 并行候选效率

对高价值任务，可以让多个 agent 同时尝试，Moxt 选择最优候选。

适用场景：小到中等复杂度 bugfix、性能优化方案、UI 实现方案、测试补全、重构策略探索。

不适用场景：大规模 schema migration、权限 / 支付 / 安全边界修改、生产事故实时修复、需求本身不清晰的任务。

### 3. 验证前置效率

每个 TaskGraph 节点创建时就生成 DeliveryContract，而不是 PR 出来后才想怎么验收。

### 4. 失败复用效率

失败不是浪费，而是训练组织调度系统：

```text
失败类型：
- task unclear
- context insufficient
- wrong agent selected
- permission too narrow
- verification incomplete
- implementation overreached
- tests misleading
```

### 5. 人类注意力压缩

人类不应审所有过程，而应只处理高风险 ActionIntent、多候选冲突、Delivery Auditor 低信心结果、影响架构方向的决策、风险预算超标任务。

## 风险与边界更新

本轮新增一个核心风险：

# Agent monoculture risk / 单一代理路径依赖

如果组织长期只用一个 coding agent，它的偏好会变成隐形架构风格。例如：

- 偏好小修小补，不敢大重构。
- 偏好通过 mock 让测试过。
- 偏好局部 patch，忽略系统一致性。
- 偏好生成更多代码，而非删除复杂度。
- 偏好满足显式任务，忽略隐性产品目标。

Moxt 应通过多 agent 比较、Delivery Auditor、Architecture Governor 和 Agent Reputation 抑制这种风险。

新增边界规则：

```text
不得自动委派的任务：
1. 需求仍含重大歧义。
2. 涉及生产数据删除。
3. 涉及权限模型重写。
4. 涉及支付、账务、合规逻辑。
5. 涉及不可逆 migration。
6. 涉及安全策略放宽。
7. DeliveryContract 无法定义。
```

## 阶段性结论

本轮最重要结论：

> Moxt 的下一阶段核心不是“更多 AI 同事”，而是“更好的 AI 委派与验证基础设施”。

推荐把产品主线收敛成 4 层：

```text
1. TaskGraph Runtime
   负责把目标变成可运行任务图。

2. Delegation Control Plane
   负责选择 agent、分配上下文、权限、预算和执行模式。

3. Verification & Delivery Layer
   负责测试、审计、候选比较和 DeliveryContract 验收。

4. Memory & Reputation Layer
   负责沉淀组织经验、agent 表现和任务模式。
```

这比单纯“AI 同事工作台”更有产品护城河，因为它把外部 agent 的能力转化为组织级工程资产。

# Agent Reputation System / 代理声誉系统

需要研究：

1. 如何给每个 agent 建立任务类型画像。

4. 如何设计 agent 选择算法。

6. 如何展示 agent 过去表现，让人类信任系统调度。
