# Moxt 进化为 AI Workflow Runtime 与研发控制面

时间：2026-05-02 12:28

## 一、本轮新增发现（关键推进）

### 1. AI 同事 ≠ 多角色，而是多阶段控制权转移

上一轮我们把系统拆成 5 个 AI 同事（需求、实现、审查、QA、发布）。本轮发现：真正的核心不是角色数量，而是“控制权在不同阶段如何转移”。

本质变化：

- 传统研发：控制权始终在人。
- Copilot 模式：控制权在人，但执行片段外包给 AI。
- Moxt 模式：控制权在流程中流动（human ↔ AI ↔ AI）。

这意味着：AI 同事设计必须绑定“控制权边界”，而不是只定义职责。

### 2. Moxt 的核心不是 workspace，而是 Workflow Runtime

之前判断为 AI Workspace，本轮修正为：

> Moxt 更接近 AI Workflow Runtime（工作流运行时）。

因为它具备：状态持续（memory）、任务持续（automation / scheduled）、多 agent 协同（AI teammates）、外部系统连接（GitHub / Slack / webhook）。

这四点组合，本质已经是一个：分布式任务调度系统 + AI 执行层。

### 3. 软件研发的关键不是写代码，而是控制不确定性扩散

新增关键判断：

> 在 AI 时代，研发系统的核心目标从“提高产出速度”转为控制 AI 引入的不确定性扩散速度。

| 传统问题 | AI 时代问题 |
|---|---|
| 人慢 | AI 太快但不稳定 |
| 人不会写 | AI 会写但不可信 |
| 交付慢 | 错误传播更快 |

因此，Moxt 的真正价值在于：把 AI 的不确定性约束在可验证边界内。

## 二、Moxt 产品理解更新（关键修正）

### 从三层到四层模型

之前：AI Team OS、Workspace、Delegation Surface。

本轮升级为：

### Moxt = AI Workflow Runtime + Delegation OS + Memory System + Governance Layer

#### 1. Workflow Runtime（新增核心层）

- 承载任务执行生命周期。
- 支持多 agent 并发。
- 管理状态流转（pending → running → verified）。

#### 2. Delegation OS

- 定义谁做什么。
- 定义任务结构（Task Packet / Spec）。

#### 3. Memory System

- 组织长期学习能力。
- 影响未来任务执行策略。

#### 4. Governance Layer（重要强化）

- 权限控制。
- 风险控制。
- 人类审批。

本轮关键变化：明确 Governance 是一等公民，而不是附属能力。

## 三、AI 同事角色设计（升级：控制权模型）

不再只是岗位，而是：

> 控制权代理（Control Agents）。

### 1. Spec Controller（需求控制者）

核心职责变化：不只是写需求，而是锁定问题空间。

新增能力：定义“不允许 AI 做什么”，定义风险范围。

关键指标：需求漂移率（Spec Drift）、后续任务重写比例。

### 2. Execution Controller（执行控制者）

变化：不直接等于 coder，而是控制实现过程的 agent。

职责：调度多个 coding agents（而不是自己写），选择策略（快 / 稳 / 低成本）。

关键新增：实现层本身可以是多 agent 竞争。

指标：最优解选择率、生成候选数 vs 最终采用数。

### 3. Verification Controller（验证控制者）

升级重点：不只是 QA，而是定义“什么叫完成”。

职责：运行验证 pipeline，判断是否满足 DeliveryContract。

新增指标：False Pass Rate（错误放行率）、Verification Coverage（验证覆盖）。

### 4. Risk Controller（新增核心角色）

职责：评估每一步操作风险，决定是否需要 human gate。

控制内容：代码变更范围、数据访问、系统影响。

指标：风险误判率、高风险拦截成功率。

### 5. Release Controller（发布控制者）

职责升级：不只是 approve，而是汇总所有 evidence，给出“可解释发布决策”。

输出：Release reasoning（为什么可以发）。

## 四、AI 原生工作空间设计（结构升级）

本轮核心升级：

> 从对象集合到状态机系统。

### 新增：Task Lifecycle 状态机

```text
Draft → Structured → Delegated → Executing → Verifying → Approved → Released → Learned
```

关键点：每一步必须可观测、可回滚、记录 evidence。

### 新增核心对象：Control Contract

之前已有 Spec、Task Packet、Rules。本轮新增：Control Contract（控制契约）。

```yaml
control_contract:
  allowed_actions:
  forbidden_actions:
  required_verifications:
  risk_budget:
  escalation_rules:
```

作用：限制 AI 行为，而不是只描述任务。

### Memory 分层（重要升级）

把 Memory 拆成三类：

1. Fact Memory：客观事实（代码结构、API）。
2. Policy Memory：团队规则（必须写测试）。
3. Outcome Memory：历史结果（某方案曾失败）。

不再允许混存，否则会污染决策。

## 五、新作业范式（进一步收敛）

本轮收敛为一句更精确的话：

> 软件研发从“任务执行系统”演化为“AI 行为约束系统”。

### 新范式核心结构

```text
Intent → Constraint → Delegation → Competition → Verification → Governance → Learning
```

关键变化：

1. Constraint（约束）成为一等输入。不是直接让 AI 干活，而是先定义边界。
2. Competition（竞争）。多个 agent 同时做，选最优。
3. Governance（治理）。不是最后一步，而是贯穿全过程。

## 六、效率提升机制（新增深层机制）

### 1. 从减少人力到增加尝试次数

AI 最大优势不是快，而是可以低成本尝试多个解。

关键指标：每任务候选解数量、最优解选择能力。

### 2. 从减少错误到限制错误影响范围

不是让 AI 不犯错，而是让错误必须被隔离在小范围，且快速发现。

### 3. 从最佳实践到动态策略选择

不同任务：有的适合快（exploration），有的适合稳（critical path）。Execution Controller 负责策略切换。

### 4. 从人 review 到机器预过滤 + 人终审

人不再看所有内容，而是只看高风险、低置信度、有冲突的结果。

## 七、风险与边界（新增关键风险）

### 1. 控制权错位（最危险问题）

如果 Execution agent 拥有过大权限，Risk Controller 不生效，会导致 AI 自主扩权（autonomous privilege escalation）。

### 2. 验证幻觉（Verification Illusion）

AI 自己生成测试并通过，会形成“伪完成”。

解决：验证必须独立 agent，或来自外部系统。

### 3. Memory 污染

错误经验进入 shared memory，会放大错误。必须引入 memory 审核，或分级可信度。

### 4. 成本失控（AI 并发问题）

多 agent 竞争会导致成本指数增长。需要 budget 控制和动态停止策略。

## 八、阶段性结论（更新版）

当前最重要的结论升级为：

> Moxt 在软件研发中的终极形态不是“AI 写代码平台”，而是一个用于“约束、调度、验证和治理 AI 行为”的研发控制系统（AI Engineering Control Plane）。

它的价值不在于让 AI 更聪明，而在于让 AI 可控、可审计、可组合。

## 九、下一轮研究方向

1. 设计 DelegationSpec × ControlContract 的统一模型。
2. 深入多 agent 竞争机制（如何选最优解）。
3. 设计 Verification Pipeline 标准结构。
4. 建立 AI 同事成本模型（credits → 工程效率）。
5. 推导 Moxt vs GitHub Actions vs CI/CD 的关系重构。
6. 深化 Risk Controller 的算法与策略设计。
