# Control Contract System（行为约束协议层）

时间：2026-05-02 21:27

## 本轮新增发现

在前几轮中系统已逐步建立完整链路：

```text
Identity → Permission → Sandbox → Execution → Verification → Evidence → Decision
```

但仍存在一个关键缺口：

> **系统可以“限制权限”，可以“验证结果”，但无法“约束行为过程”。**

具体表现：

```text
- Permission 只定义“可以做什么”，但不定义“必须怎么做”
- Verification 只在事后检查，而不是过程控制
- Agent 在执行中可能：
  - 跳过中间步骤
  - 使用不推荐策略
  - 生成 technically correct 但不符合组织规范的方案
```

本轮关键新增：

> **Control Contract System（行为约束协议层）**

## MOTX/Moxt 产品理解更新

系统从“决策系统”进一步升级为：

```text
Moxt = Constraint-Driven AI Engineering System
```

新增核心层：

```text
Layer 0: Control Contract（行为约束）
Layer 1: Identity & Permission
Layer 2: Sandbox Execution
Layer 3: Verification Pipeline
Layer 4: Evidence Modeling
Layer 5: Decision Engine
```

关键认知升级：

> 仅靠“权限 + 验证”是不够的，必须在执行前定义“行为契约”。

## AI 同事角色设计更新

新增核心角色：

### Control Contract Designer / 行为约束设计 AI

**定位**：为每个任务生成行为约束协议（Control Contract），定义 agent 在执行过程中必须遵守的规则。

### 职责

- 根据 Task 类型、RiskProfile、Repo 区域生成 ControlContract；
- 定义 required steps、forbidden patterns、required tools、required intermediate artifacts；
- 定义执行约束：是否必须写 test、是否必须先生成 design、是否必须跑某些分析工具；
- 定义行为 guardrail：不允许直接修改某些文件、不允许引入某类依赖；
- 与 VerificationPipeline 对齐，保证“过程约束”和“结果验证”一致。

### 工具权限

- 可读取 TaskGraph、RiskProfile、Repo policy；
- 可写 ControlContract；
- 不可修改代码；
- 不可执行任务。

### 协作边界

- 不执行代码（Worker）；
- 不验证结果（Verification）；
- 不做决策（Decision Engine）；
- 只定义“行为规则”。

### 评价指标

```text
contract violation rate
prevented risk incidents
over-constraint rate（过度约束）
execution efficiency impact
alignment with org standards
```

## 软件研发新作业范式更新

范式再次升级为：

> **Contract-Constrained, Decision-Driven Agentic Development**

完整流程：

```text
1. IntentSpec
2. ChangePlan
3. ControlContract
4. Agent Identity & Permission
5. Sandbox Execution
6. BehaviorTrace
7. Verification Pipeline
8. EvidencePack
9. Decision Engine
```

### 核心变化

#### 1. 从结果导向到过程 + 结果双约束

过去只要 PR 正确即可。现在必须按正确方式产生 PR。

#### 2. Agent 行为可控

系统可以约束执行顺序、工具使用、中间产物、策略选择。

#### 3. Contract 与 Verification 联动

```text
Contract 定义必须做什么
Verification 检查是否真的做了
```

## 效率提升机制

### 1. 减少无效尝试

约束 agent，避免走错误路径，减少 trial-and-error。

### 2. 提高结果一致性

不同 agent 遵守同一 contract，输出更稳定。

### 3. 降低 review 成本

因为过程已被约束，reviewer 不需要猜 agent 做了什么。

### 4. 支持组织规范自动化

把 best practice、coding standard、安全规则变成 machine-enforced contract。

## 风险与边界

### 1. 过度约束（Over-Constraint）

过度约束可能限制 agent 创造力，降低问题解决能力。

### 2. Contract 设计复杂度

需要为不同任务设计不同 contract。

### 3. 动态任务不适配

复杂任务无法预定义所有步骤。

### 4. 执行开销

实时检查会增加系统复杂度。

## 阶段性结论

Moxt 系统已经演化为：

```text
Intent
→ Plan
→ Contract
→ Identity
→ Permission
→ Sandbox
→ Execution
→ Verification
→ Evidence
→ Decision
```

关键认知升级：

> 软件研发中的 AI 控制问题，本质是“在不确定执行中施加确定性约束”。

最终系统本质：

> **Moxt = Constraint + Execution + Verification + Decision 的统一控制系统**
