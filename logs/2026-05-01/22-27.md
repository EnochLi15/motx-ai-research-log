# MOTX/Moxt 软件研发范式研究增量记录

时间：2026-05-01 22:27

## 本轮新增发现

1. AI coding agent 的产品形态正在进一步分化为两条线：
   - 单人开发者侧：IDE、CLI、桌面 app、云端 sandbox，用于快速完成局部任务；
   - 组织侧：多 agent 指挥台、权限治理、PR/CI/发布门禁、审计与指标，用于让 AI 进入真实研发组织。

2. GitHub Copilot coding agent 已经把“从 issue 到 PR”做成平台级能力：用户可以从 GitHub Issues、agents panel、GitHub CLI、IDE、GitHub MCP server 等入口把任务交给 Copilot；Copilot 在后台工作、提交 draft PR，并请求人类 review。这说明行业默认工作流正在从“人写代码 + AI 辅助”转向“人派发任务 + AI 提交可审查产物”。

3. OpenAI Codex 的公开定位从“云端软件工程 agent”演进为“多 agent command center”：支持多个 agent 并行、worktree 隔离、云端环境、长期任务、自动化触发。这强化了 Moxt 的判断：未来开发者不是只和一个 chat assistant 对话，而是在监督多个可执行 AI worker。

4. SWE-bench Verified、Vexp SWE-bench、Terminal-Bench 等 benchmark 越来越强调真实工程执行能力：resolution rate、cost per task、duration、token usage、是否能在终端完成多步骤任务。这说明 Moxt 的评价体系不能只看“生成代码是否正确”，而要看端到端吞吐、成本、风险和组织协作质量。

5. PocketOS 数据删除事故继续提供强信号：AI agent 的核心风险不是“模型会不会道歉”，而是“系统是否允许它在没有结构化意图、审批、权限隔离、备份隔离的情况下执行不可逆动作”。Moxt 的风险控制必须前置到工具调用之前，而不是依赖 agent prompt 中的安全规则。

## MOTX/Moxt 产品理解更新

Moxt 应避免与 Cursor、Claude Code、Codex、Copilot 在“谁写代码更快”上正面对抗。更合适的产品定位是：

**AI software organization control plane**

也就是为企业研发团队提供：

- AI 同事身份与职责管理；
- 任务派发与状态机；
- 多 agent worktree / sandbox 编排；
- 研发对象图谱；
- 工具权限和环境边界；
- PR、CI、Release 的门禁控制；
- 成本、质量、风险、吞吐量指标；
- 审计、回滚和知识沉淀。

Moxt 的差异化不应该是替代现有 coding agent，而是把它们纳入组织级工作流：一个团队可以同时使用 Copilot、Codex、Claude Code、Cursor、内部 agent，但所有 AI 行为都必须进入 Moxt 的对象协议、权限系统和审计链路。

## AI 同事角色设计更新

本轮建议在 8 个 AI 同事基础上增加一个横向概念：**Agent Adapter / 工具型外部 agent 适配器**。

它不是第 9 个 AI 同事，而是 Moxt 连接外部 coding agent 的抽象层。

### Agent Adapter

职责：

- 将 GitHub Copilot、Codex、Claude Code、Cursor、内部脚本 agent 的任务状态统一映射为 Moxt Task / ActionIntent / Patch / TestRun；
- 收集 agent session log、diff、命令、测试结果、成本与耗时；
- 把外部 agent 的能力声明成 Capability Manifest；
- 把 Moxt 的权限策略翻译成外部工具可执行的限制。

边界：

- 不直接做产品决策；
- 不直接授予生产权限；
- 不绕过 Security Sentinel；
- 不把外部 agent 的“成功”直接等同于可发布。

评价指标：

- 外部 agent 状态同步准确率；
- 任务映射完整度；
- session log 可追溯率；
- 成本归因准确率；
- 权限策略执行一致性。

### Coordinator 更新

Coordinator 需要从“分派给内部 AI 同事”升级为“调度内部 AI 同事 + 外部 agent”。它应基于任务类型选择执行器：

- 低风险小修复：Copilot / Codex / Claude Code 可直接生成 PR；
- 复杂架构任务：Tech Lead 先拆解，再分派多个 Implementation Engineer；
- 安全敏感任务：Security Sentinel 先审 ActionIntent；
- 大规模重构：必须使用 worktree 隔离、模块锁、测试计划和分阶段 PR。

### Reviewer 更新

Reviewer 不只审代码，还要审 agent 行为：

- 是否遵守任务边界；
- 是否修改了非预期文件；
- 是否引入高风险依赖；
- 是否绕过测试；
- 是否出现“为了让测试通过而弱化测试”的行为。

### QA / Release Guardian 更新

QA 应新增“agent-produced PR risk profile”：

- AI 生成代码比例；
- 涉及关键路径数量；
- 测试覆盖变化；
- 失败测试是否被删除或跳过；
- 是否需要 human exploratory testing。

## AI 原生工作空间设计

本轮建议新增三个对象：`AgentSession`、`CapabilityManifest`、`RiskBudget`。

### AgentSession

```yaml
AgentSession:
  id: session_001
  agent_provider: codex
  agent_identity: external_codex_worker_7
  task_id: task_123
  sandbox_id: sandbox_456
  branch: ai/task-123
  started_at: 2026-05-01T22:00:00+08:00
  ended_at: 2026-05-01T22:18:00+08:00
  commands_executed:
    - npm test
    - npm run lint
  files_changed:
    - src/billing/reconcile.ts
    - tests/billing/reconcile.test.ts
  tokens_used: 184000
  estimated_cost_usd: 3.42
  status: proposed_patch
  linked_patch: patch_789
  risk_flags:
    - touches_billing_logic
    - modifies_migration_file
```

### CapabilityManifest

```yaml
CapabilityManifest:
  agent_provider: github_copilot_coding_agent
  supported_inputs:
    - github_issue
    - prompt
    - pull_request_comment
  supported_outputs:
    - draft_pull_request
    - commit
    - session_log
  can_run_tests: true
  can_access_network: configurable
  can_access_secrets: false_by_default
  requires_human_before_ci: true
  max_autonomy_level: L3
```

### RiskBudget

```yaml
RiskBudget:
  scope: team_checkout
  period: weekly
  max_high_risk_actions: 3
  max_parallel_agents: 8
  max_unreviewed_ai_prs: 5
  max_cost_usd: 500
  required_escalation:
    - production_data_access
    - schema_migration
    - auth_logic_change
    - payment_logic_change
```

Moxt 工作空间应将外部 agent 的输出全部纳入统一对象链路：

```text
Goal
 -> Requirement
 -> Task
 -> AgentSession
 -> ActionIntent
 -> Patch
 -> TestRun
 -> Review
 -> ReleaseCandidate
 -> Release
 -> Observation
 -> Memory
```

## 软件研发新作业范式

本轮提出：**Agent Swarm Review Loop / AI 群体执行-审查闭环**。

新流程：

1. 人类提出 Goal。
2. Product Analyst 和 Tech Lead 生成 Requirement / Design / Task。
3. Coordinator 根据任务风险和耦合度拆分执行计划。
4. Agent Adapter 选择外部或内部 coding agent，并创建 AgentSession。
5. Implementation Engineer 或外部 agent 在隔离 worktree / sandbox 中生成 Patch。
6. Reviewer 审代码和 agent 行为。
7. QA 生成风险画像并运行测试矩阵。
8. Security Sentinel 审 ActionIntent 和敏感 diff。
9. Release Guardian 决定是否进入 release candidate。
10. Knowledge Steward 将稳定经验写入 Approved Memory。

关键区别：Moxt 不要求所有 AI 都在同一个模型或同一个工具里，而是把所有 AI 输出收敛到统一研发对象图谱和治理流程中。

## 效率提升机制

本轮新增三个效率杠杆：

### 1. Agent 选择路由

不同任务分配给最合适的执行器：

- 简单 bug：低成本 agent；
- 复杂 refactor：强推理 agent；
- 文档与测试：低风险自动化 agent；
- 安全敏感代码：先由 Sentinel 生成约束，再执行。

指标：cost per accepted change、first-pass merge rate、agent-task fit score。

### 2. Worktree 并行 + 模块锁

允许多个 agent 同时处理同一 repo 的不同区域，但必须声明文件/模块锁，避免互相覆盖。

指标：parallel task throughput、merge conflict rate、stale branch rate。

### 3. AI PR 风险分层

不是所有 AI PR 都用同样的人类 review 强度。Moxt 可按风险画像决定：

- 低风险：轻量 review；
- 中风险：常规 review + 测试；
- 高风险：架构 review + 安全 review + QA sign-off；
- 极高风险：人类手工执行或双人审批。

指标：human review minutes saved、defect escape rate、review queue latency。

## 风险与边界

本轮新增风险：

1. **外部 agent 黑箱化**：外部工具可能只返回 PR，不返回完整决策过程和命令历史。Moxt 必须要求 session log 或至少记录 diff、命令、测试、成本。

2. **Agent 之间相互放大错误**：一个 agent 生成错误假设，另一个 agent 基于该假设继续实现。需要 Knowledge Steward 管理 Memory 的置信度和审批状态。

3. **Benchmark 误导**：高 SWE-bench 分数不代表适合企业代码库。Moxt 需要组织内 benchmark：历史 bug、真实 PR、内部模块、权限限制下的成功率。

4. **成本失控**：多 agent 并行会把 token 和云资源成本放大。RiskBudget 应同时限制风险动作和成本。

5. **CI/CD 权限链路被间接触发**：即使 agent 不能直接部署，也可能通过 PR 触发 workflow。需要结合 branch protection、CODEOWNERS、workflow approval 和 secret policy。

默认边界建议：

- 外部 agent 默认不能访问生产 secret；
- 外部 agent 输出必须进入 draft PR；
- 高风险 PR 不自动运行有 secret 的 CI；
- migration、auth、billing、permission、infra 目录默认触发 Security Sentinel；
- agent session 超出 RiskBudget 自动暂停。

## 阶段性结论

Moxt 的关键机会是成为“多 AI 工程组织的控制面”。

随着 Copilot、Codex、Claude Code、Cursor 等工具都在增强“可委托工程执行”，单个工具的编码能力会快速商品化。企业真正缺的是：

```text
谁可以派发任务
派给哪个 agent
agent 能访问什么
agent 做了什么
成本是多少
风险是多少
谁审过
如何回滚
哪些经验能沉淀为组织记忆
```

本轮结论：Moxt 应把外部 coding agent 视为可插拔 worker，而不是竞争对象；Moxt 的核心资产是研发对象协议、AgentSession 记录、Risk Control Plane、AI PR 风险分层和组织级效率指标。

## 下一轮研究方向

1. 设计 Agent Adapter 的标准接口：输入、输出、状态、日志、错误、成本。
2. 设计 AI PR Risk Profile 的评分模型。
3. 设计组织内 benchmark：用历史 issue/PR 测试不同 agent 的真实成功率和成本。
4. 研究 Moxt 如何与 GitHub branch protection、CODEOWNERS、workflow approval、secret scanning 联动。
5. 细化 RiskBudget 与 ActionIntent 的关系：预算如何影响 agent 自治等级和工具权限。
