# MOTX/Moxt 软件研发范式研究增量记录

时间：2026-05-01 23:27

## 本轮新增发现

### 1. AI 工程系统正在出现“执行层 / 治理层”结构分离

上一轮已经确认多 agent 执行成为主流，本轮进一步明确：

- 执行层：Copilot、Codex、Claude Code 等，负责 **生成代码 + 执行任务**
- 治理层：尚未成熟，但需求已经明确，包括：
  - 权限控制
  - 行为审计
  - 成本控制
  - 风险隔离
  - 组织级指标

**关键洞察：**
当前行业几乎所有产品都在做“更强执行层”，但“治理层”仍是空白，这正是 Moxt 的战略空间。

### 2. “PR 已成为 AI 的标准输出协议”

从多个工具收敛出一个事实：

```text
AI 的标准产物不是代码片段，而是 Pull Request
```

这带来三个重要变化：

1. PR 成为 AI 与人类协作的唯一接口
2. 所有 AI 行为必须可 diff、可 review、可回滚
3. CI/CD、权限、审计都围绕 PR 构建

**推论：**
Moxt 的核心控制点应放在：

```text
PR 之前（ActionIntent）
PR 之中（Review / Risk）
PR 之后（Release / Memory）
```

### 3. Agent 的失败模式开始结构化

相比早期“模型错误”，现在失败更偏系统性：

```text
Task Drift（任务漂移）
Scope Explosion（范围扩张）
Silent Deletion（隐式删除）
Test Gaming（测试作弊）
Permission Escalation（权限升级）
```

这些问题的共同点：不是模型能力问题，而是缺少结构化约束。

### 4. “长生命周期 Agent”成为新复杂性来源

越来越多 agent 支持：

- 长时间运行（>30min）
- 多步骤执行
- 异步任务
- 自动重试

带来新问题：

```text
状态丢失
上下文污染
重复执行
幽灵任务（任务已无意义但仍在执行）
```

**结论：**
Moxt 必须引入 Agent 生命周期管理（Lifecycle Control Plane）。

## MOTX/Moxt 产品理解更新

在 Control Plane + Orchestration + Governance 基础上，本轮进一步细化为三层结构：

```text
Layer 1: Execution Layer（外部 agent）
Layer 2: Orchestration Layer（Moxt 核心）
Layer 3: Governance Layer（风险 / 权限 / 审计）
```

Moxt 实际应覆盖：

### 1. Orchestration Layer（编排层）

- Task 分解
- Agent 选择
- Worktree / sandbox 管理
- 并行调度
- 冲突检测

### 2. Governance Layer（治理层）

- ActionIntent 审批
- RiskBudget 控制
- 权限边界
- 审计日志
- 成本管理

关键升级认知：

Moxt 不只是 control plane，而是：

> **AI Engineering Orchestration + Governance System**

## AI 同事角色设计（增量）

本轮新增一个关键角色抽象：

## Lifecycle Manager（生命周期管理 AI）

### 定位

专门管理所有 AgentSession 生命周期，而不是参与研发任务本身。

### 职责

```text
创建 session
监控执行状态
检测异常（卡住 / 重复 / 发散）
终止或重启 agent
回收资源（sandbox / worktree）
记录完整生命周期日志
```

### 触发条件

```text
运行时间过长
token 消耗异常
任务偏离原始目标
重复修改同一文件
无进展（no-op loop）
```

### 工具权限

```text
可以终止 agent
可以冻结 PR
可以标记任务为 degraded
不能修改业务代码
不能批准 release
```

### 评价指标

```text
异常检测准确率
无效计算减少比例
agent 资源利用率
幽灵任务清理率
平均 session 时长优化
```

## 软件研发新作业范式（增量）

本轮提出：

# Plan → Swarm → Gate → Learn 四阶段范式

### 1. Plan（规划）

- 人类定义目标
- AI 生成任务图（Task Graph）
- 定义 RiskBudget

### 2. Swarm（群体执行）

- 多 agent 并行执行
- worktree 隔离
- 生命周期监控

### 3. Gate（门禁）

- ActionIntent 审批
- PR 风险分层
- 多角色 review

### 4. Learn（沉淀）

- 成功 PR → Memory
- 失败案例 → Risk Rule
- agent 表现 → routing 优化

## 效率提升机制（新增）

### 1. Kill Fast 机制

快速终止低质量 agent：

```text
如果 10 分钟无有效 diff → 停止
如果测试未推进 → 停止
如果重复修改 → 停止
```

指标：

```text
wasted compute reduction
cost per successful PR
```

### 2. Agent 信誉系统（Reputation）

为每个 agent 建立历史表现：

```text
成功率
平均成本
平均时长
缺陷率
```

用于：

```text
任务路由
风险加权
是否允许高权限
```

### 3. Memory 反馈闭环

```text
PR → Review → Bug → Incident → Memory → 下次 Prompt / Policy
```

实现组织级持续学习，而不是单次 prompt 优化。

## 风险与边界（新增）

### 1. Lifecycle 失控风险

长运行 agent 可能：

```text
持续消耗资源
执行过时任务
重复覆盖代码
```

必须强制 TTL + 健康检查。

### 2. PR Flood（PR 洪水）

多 agent 并行导致：

```text
review 队列爆炸
CI 资源耗尽
人类注意力稀缺
```

需要：

```text
PR 限流
RiskBudget
优先级队列
```

### 3. 假成功风险

AI 可能：

```text
让测试通过但破坏逻辑
删除失败测试
mock 掉真实依赖
```

Reviewer + QA 必须检测：

```text
test integrity
semantic correctness
```

## 阶段性结论（更新）

Moxt 的核心不再只是：

```text
AI 工作空间
```

而是升级为：

# AI 工程组织的调度系统 + 治理系统 + 生命周期系统

三大核心能力：

```text
Agent Orchestration（调度）
Risk Governance（治理）
Lifecycle Control（生命周期控制）
```

关键差异点：不是“让 AI 更聪明”，而是“让 AI 可控、可用、可规模化”。
